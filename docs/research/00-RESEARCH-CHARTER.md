# Deltawright enhancer research — charter

**Goal:** figure out how to make Deltawright a *genuine* enhancer for Playwright test authoring —
giving authors/agents something Playwright does **not** provide — instead of a narrow diagnosis tool.
Research DEEPLY first (this folder), then design, then a targeted prototype. No blind experimentation.

## What DW is (context for every research block)
DW = an action-scoped **delta + actionability** layer on Playwright (ESM-only npm `deltawright`). Existing
primitives (the building blocks the research should compose, not ignore):
- **`actAndObserve(page, action)`** → a compact, coalesced **delta of what changed** after ONE action, each
  node geometry-annotated + actionability-reconciled against Playwright's authoritative verdict.
- **Geometry reads** — `probeGeometry`/`geometryVerdict`: rect, `coveredBy` (what overlays it), `offscreen`,
  `hitSelf`, z-order. (DW already knows *where things are and what covers what*.)
- **Injected MutationObserver + `awaitQuiescence`** — a structural + network settle SIGNAL (bounded).
- **`diffChangedRegion` / screenshot-diff** — a **pixel-region change** primitive (the literal "screenshot").
- **`preflight`/`toBeActionable`** — Playwright's actionability verdict + a geometry-disagreement hint.
- **input-integrity** (committed-value drift), **routeSignals** (backend/app/DOM co-occurrence routing),
  **`suggest()`** (selector/assertion candidates from a delta), closed **taxonomy + `diagnose`**.
- Honesty contracts: observe/label, **never fabricate/retry/suppress**; Playwright verdict authoritative;
  co-occurrence never causation.

## The verdict that motivates this (from ~a week of dogfooding experiments)
Across a diagnosis demo, a test-authoring study, a nightly-flake RCA, and a suite-hardening run on a real
legacy-GWT enterprise Playwright suite, the consistent finding: DW is a **strong diagnosis/triage layer**
but **narrow as an authoring/hardening enhancer** — most "better/faster tests" is plain Playwright
discipline DW doesn't own. To matter long-term, DW must offer something **categorically different** from
Playwright. (Full findings + backlog: `00-FINDINGS-BACKLOG.md`.)

## The vision (what "genuine enhancer" means here)
A human reads a **screenshot** and instantly sees state — positions, what covers what, what's ready, what
changed. Playwright instead serializes a **DOM/a11y snapshot** and relies on **WAITS** to make actions
reflect before acting. **DW should give the richer "picture" + a wait-free readiness signal.** Concretely:
1. **Selector + positioning/geometry mapping as first-class DW methods** — a spatial+semantic page model.
2. **A wait-free readiness mechanism** — know an action has "reflected" WITHOUT static sleep OR networkidle.
3. Net: Playwright tests that are **better, smoother, richer** to write.

## Hard constraints (every research block must honor)
- **Do NOT copy Playwright methods** (`waitForSelector`/`networkidle`/`expect`) and reship as DW. Provide
  what Playwright LACKS. If a proposed idea is "networkidle rebranded" or "waitForSelector with a nicer
  name," say so and discard it.
- Must be **genuinely differentiated** (the screenshot-vs-snapshot gap).
- Keep DW's honesty contracts. Prefer composing DW's existing primitives (geometry, delta, screenshot-diff,
  observer) over inventing from scratch.

## Research blocks in this folder
- `01-wait-free-readiness.md` — R1 (THE crux): readiness without temporal waits or global networkidle.
- `02-spatial-page-model.md` — R2: the "screenshot-equivalent" page representation for authoring/agents.
- `03-durable-selectors.md` — R3: unique+stable selector intelligence from the live DOM+geometry.
- `04-competitive-gap.md` — R4: how other tools solve auto-wait/page-understanding/selectors → the unmet gap.
- `META-SYNTHESIS.md` — cross-block synthesis + a recommended direction to replan from.

Each block: prior-art survey (cited where possible) → candidate approaches → how DW's primitives enable it →
**honest differentiation check (real vs Playwright-rebranded)** → feasibility/risks → recommended direction
+ a minimal prototype sketch.

## How this research was run (and how to re-run / continue it)
Each block report was generated by an independent research pass with web access, given: this charter's DW
context + vision + constraints, that block's question (see `00-FINDINGS-BACKLOG.md` §E, R1–R4), and the
instruction to survey prior art (cited URLs), do an honest real-vs-Playwright-rebranded check, and write the
block's `0X-*.md` file. The reports are self-contained on disk. **To continue:** if a block file is missing
or thin, re-run that block from this charter + its question (no hidden state is needed). Once all four blocks
exist, write `META-SYNTHESIS.md` (cross-block synthesis → the single most defensible direction) and replan
from it — do not start prototyping before the synthesis.
