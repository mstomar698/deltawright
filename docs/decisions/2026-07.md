## 2026-07-07 — Build on Playwright and reuse its actionability engine, do not reimplement it

**Decision:** Deltawright stands on Playwright + headless Chromium. The delta's
actionability verdict is obtained from Playwright's own engine via
`locator.click({ trial: true })` (full actionability checks, no click performed).
Our geometry read is an explanatory annotation only; Playwright's verdict is
authoritative and wins any disagreement.

**Context:** The core failure being fixed is "present in the DOM but not
actionable." If we invented a parallel actionability judgment it would drift from
what a real Playwright action does — recreating the very gap we are closing.

**Alternatives rejected:**
- Reconstruct actionability from raw CDP DOM/overlay domains — reimplements Playwright, high drift risk.
- Geometry-only verdict (rect + `elementFromPoint`) — misses stability/enabled/receives-events nuances Playwright already encodes.

**Affected areas:** `src/host/actionability.ts`, `src/host/actAndObserve.ts`.

**Tags:** `architecture` `contract`

## 2026-07-07 — Delta membership is mutation-derived; geometry is annotation, never a filter

**Decision:** The set of reported nodes comes only from coalesced MutationRecords
(net added/removed/attr/text). `getBoundingClientRect` / `getComputedStyle` /
`elementFromPoint` only annotate a node; they never remove it from the delta.

**Context:** Off-screen and covered nodes are exactly the ones we must report (with
a NOT-actionable verdict). Using geometry as a membership filter would silently
drop the two most important cases.

**Alternatives rejected:**
- Filter out non-topmost / off-viewport nodes to "reduce noise" — would drop the covered/off-screen deliverables.

**Affected areas:** `src/injected/observer.ts` (`coalesce`, `readGeometry`, `collect`).

**Tags:** `architecture` `pattern`

## 2026-07-07 — v0.1 settle detection is a deliberately simple, labeled quiescence heuristic

**Decision:** Settle = "at least one mutation seen AND DOM quiet for `quietMs`, OR
`maxWaitMs` elapsed", then await running animations (`getAnimations().finished`,
capped by `animMaxMs`) before reading geometry. Defaults `{120, 2000, 1000}` ms.
The quiet timer never fires before the first mutation (defeats the delayed-insert
trap). This is explicitly a v0.1 heuristic, hardened in v0.5.

**Context:** Settle is the #1 risk (§6). CSS transitions move geometry without
firing mutations, so quiescence alone is insufficient — hence the animation await.

**Alternatives rejected:**
- Fixed sleep — brittle and slow.
- Network-idle only — misses pure-DOM/animated changes.

**Affected areas:** `src/injected/observer.ts` (`waitForSettle`, `settleAnimations`). See DW-01.

**Tags:** `architecture` `performance`

## 2026-07-07 — v0.1 actionability verdict is pointer/click-actionability, probed via click trial

**Decision:** The delta's `verdict` answers "can this be clicked?" and is obtained
from a single probe, `locator.click({ trial: true })`, for every node. This matches
the design doc's model (a covered control is NOT-actionable). Role-aware probes
(`fill` editability for text inputs, `selectOption`, `hover`) are deferred to v0.5.

**Context:** An adversarial verification noted `click({trial})` is wrong in both
directions for non-click actions: a covered text input is reported NOT-actionable
though `fill` (no hit-test) would succeed; a readonly input is reported ACTIONABLE
though `fill` throws. For v0.1 we choose one consistent, honestly-labeled semantic
rather than a half-built role dispatch.

**Alternatives rejected:**
- Per-role probe dispatch now — real v0.5 work; premature and under-tested at v0.1.
- Leave "ACTIONABLE" unqualified — misleads for fill/select/hover.

**Affected areas:** `src/host/actionability.ts`, `src/host/types.ts`. See DW-02.

**Tags:** `contract` `architecture`

## 2026-07-08 — #30 attribution stays baseline-authored; the trusted-event anchor is KEEP-ONLY; independent in-window dropping is declined

**Decision:** For #30 (advanced causal attribution) the pre-arm baseline remains the
**sole authority** for dropping background element-insertions (a signature that recurred
≥2× *before* the action). The new trusted-event anchor (`inWindowRecurrence: true`,
default off) is **KEEP-ONLY**: it makes that drop per-root, rescuing the action's own
instance of a background-looking signature when it is DOM-local to, or within
`ANCHOR_RADIUS = 64 px` of, the trusted click. It can never cause a drop and never
promote a signature to background. **Independent in-window dropping** (condemning a
signature on within-window evidence alone, to remove the ~150 ms baseline latency) is
**declined as unsafe** — including the weaker "baseline-count ≥ 1 reinforces" variant.

**Context:** A 4-design adversarial panel (temporal-cadence, locality+recurrence,
signal-fusion, skeptic-minimal) reached a unanimous, adversary-verified result: with only
within-window signal you cannot separate a long causal payload (streamed search results,
a staggered master-detail mount, an SSE generate-stream longer than the settle cap, a
multi-wave refresh) from a recurring background feed — they share signature, cadence,
temporal spread, and a truncated tail, and `maxWaitMs` is a timeout, not a semantic
boundary, so "still arriving at the cap" is information-free. Every independent design had
a high-danger, unclosable **false-drop** (dropping the action's real change — the one
unforgivable error under the "when uncertain, KEEP" invariant). Even baseline-count == 1
corroboration breaks, because `insertSig` keys on `classList[0]` (a shared Tailwind/hashed
class), so count == 1 collisions between an unrelated baseline insert and the action's real
output are common. The anchor rescue, by contrast, only ever *keeps*, survives every panel
scenario trivially (none of those payload signatures are in `bgInsert`), and fixes a latent
false-drop already in the shipped slice (an app reusing a toast class for its own
confirmation).

**Alternatives rejected:**
- Independent in-window drop via cadence/CoV/tail/group-count/far-from-anchor scoring — each adversarially shown to false-drop a real streamed or staggered payload.
- Baseline-count ≥ 1 "reinforcement-only" promotion — breaks on shared-class signature collisions (the CRM-refresh case drops a whole repopulated dashboard).
- Cutting `baselineMs` and recovering the lost drops in-window — the recovery signal is exactly the unsafe one above.

**Affected areas:** `src/injected/observer.ts` (anchor capture, `isActionLocal`, per-root
coalesce), `src/host/actAndObserve.ts` (`inWindowRecurrence`). Tests `test/rescue.spec.ts`.
Full rationale in `docs/summaries/v0.5-inwindow-attribution-decision.md`.

**Tags:** `architecture` `contract`

## 2026-07-08 — Package as a self-contained ESM build; Playwright is a peer, esbuild is not shipped

**Decision:** `npm run build` (tsup + a small `scripts/build.mjs`) emits `dist/`: ESM +
bundled `.d.ts` for the public entry points, executable `deltawright`/`deltawright-mcp`
bins, and a **pre-bundled injected-observer IIFE** (`dist/injected/observer.global.js`) that
the host reads at runtime. `package.json` exports/main/types/bin point at `dist/`; `files`
ships only `dist` + docs; `prepack` builds so publishes are always fresh. `@playwright/test`
is a **peerDependency** (consumers bring their own; the core uses type-only imports),
`gpt-tokenizer` is a runtime **dependency**, and `esbuild` stays a **devDependency**.

**Context:** Before this, `exports` pointed at raw `./src/*.ts` run via tsx, so nothing was
installable and every planned wrapper (reporter/matchers/wait) was un-shippable. Two latent
faults would have made a naive `dist/` install-but-not-run: `inject.ts` esbuild-bundled the
observer from `../injected/observer.ts` **at runtime** (that `.ts` is not in a bundled
`dist/`), and both `esbuild` and `gpt-tokenizer` — real runtime imports — sat in
`devDependencies`.

**Alternatives rejected:**
- Ship `src/` and keep runtime esbuild bundling of the observer — makes esbuild a heavy
  runtime dependency and couples the installed package to a source-tree layout.
- Raw `tsc` per-file `.d.ts` emit — under nodenext a consumer needs `.js` extensions in the
  emitted declarations' relative imports; tsup's bundled, self-contained `.d.ts` sidesteps it.
- Declare subpath exports now for not-yet-built capabilities (`./reporter`, `./matchers`,
  `./wait`, `./aggregate`) — would resolve to missing files and break consumers/typecheck.
  Each capability ticket adds its own subpath entry when it ships. Only `.` and `./mcp` exist.

**Notes:** The `DeltawrightApi` interchange type moved to `src/host/types.ts`; the host
reads `window.__deltawright` via a local cast instead of the injected module's global
augmentation, so the shipped `.d.ts` neither depends on the injected file nor augments a
consumer's global `Window`. Packaging is locked by `test/packaging.spec.ts` (built-package
import under plain node, MCP bin from `dist/`, by-name consumer typecheck under nodenext,
byte-for-byte default-serializer parity src vs. build).

**Affected areas:** `package.json`, `scripts/build.mjs`, `src/host/inject.ts`, `src/cli.ts`,
`src/host/types.ts`, `src/host/actAndObserve.ts`, `src/injected/observer.ts`, `tsconfig.json`,
`.github/workflows/ci.yml`, `test/packaging.spec.ts`.

## 2026-07-08 — One closed, versioned root-cause taxonomy (DW-04)

**Decision:** All v0.6 diagnosis is expressed in ONE closed enum of 18 root-cause codes
(`src/host/taxonomy.ts`) across six categories, with `unknown` as a first-class "unsure"
outcome. Each code is grounded in ≥1 real `PrimitiveSignal` (a concrete injected-read /
settle-result / Playwright-engine output); `catch-all` is reserved for `unknown`. The set
is versioned and change-gated by **DW-04**: adding/removing/renaming a code requires an ADR,
a corpus relabel (#51), and an accuracy-harness re-run (#52), then updating the frozen SHA
lock in `test/taxonomy.spec.ts` as the deliberate last step.

**Context:** Six planned diagnosis surfaces (pre-flight matcher, side-car, MCP assist,
authoring aid, aggregator, and the engine itself) would otherwise each grow their own
strings for the same causes, so the "single accurate place" fractures into dialects. An
ungrounded or ad-hoc code is exactly how a confidently-wrong label ships — the product's
named anti-goal. Closing and grounding the vocabulary up front, before any diagnosis is
emitted (#48), makes drift a test failure rather than a silent regression.

**Alternatives rejected:**
- Let each surface name causes ad hoc, reconcile later — guarantees drift and a relabel
  churn once the corpus (#51) exists.
- An open/extensible enum — removes the forcing function; a new code could ship without an
  ADR, corpus relabel, or harness re-run, and without a grounding signal.
- Ground codes against live type fields today — the gap flags (`lateStructural` #49,
  `stable` #50) don't exist yet; the taxonomy is the CONTRACT those tickets fulfil, so it
  legitimately names forward signals, validated against a closed `PrimitiveSignal` set.

**Affected areas:** `src/host/taxonomy.ts`, `src/index.ts`,
`docs/specs/v0.6-root-cause-taxonomy.md`, `docs/decisions/design-watches.md` (DW-04),
`test/taxonomy.spec.ts`.

**Tags:** `accuracy` `taxonomy` `contract`

## 2026-07-08 — One Confidence primitive; unsure is first-class (DW-03)

**Decision:** Every v0.6 diagnosis carries one shared `Confidence`
(`confirmed | suspected | unknown`, `src/host/confidence.ts`). `confirmed` is earned ONLY
when an authoritative engine named the cause — a Playwright error/verdict, or a
geometry+Playwright agreement; a geometry-only or timing-only read is at most `suspected`;
insufficient evidence is `unknown`; and conflicting signals downgrade one notch, never up.
`assessConfidence(evidence)` is the single owner of these rules. A diagnosis is a hypothesis
about *why* and must never contradict Playwright's actionability verdict (DW-02/DW-03).

**Context:** "Unsure beats confidently wrong" is the product's named anti-goal defense.
Three draft proposals each re-declared a confidence notion, which is how the highest-volume
flake mode ends up mislabeled with false certainty. Collapsing to one engine-owned type,
with `unknown` as a first-class output rather than an embarrassing fallback, makes silence
the default when evidence is thin.

**Alternatives rejected:**
- A boolean confident/not — loses the crucial middle band (`suspected`) where geometry or
  timing hints something without an authoritative engine confirming it.
- Let geometry-only reads claim `confirmed` when they "look obvious" — exactly the confident
  wrong-label failure; geometry is annotation, Playwright is authoritative (DW-02).
- Fold confidence into the taxonomy code — couples "what" to "how sure", so the same cause
  can't be reported confirmed on one page and suspected on another.

**Affected areas:** `src/host/confidence.ts`, `src/index.ts`,
`docs/decisions/design-watches.md` (DW-03), `test/confidence.spec.ts`.

**Tags:** `accuracy` `confidence` `contract`

## 2026-07-08 — diagnose(delta) is agree-or-flag; the verdict is never overridden

**Decision:** Root-cause diagnosis is ONE pure function `diagnose(delta): DiagnosedDelta`
(`src/host/diagnose.ts`) that reads only the existing delta/stats/actionability and attaches
`Diagnosis[]`. Core rule — AGREE-OR-FLAG: a NOT-actionable node earns a specific
specific cause code ONLY from the branch where both engines already agree it is blocked
(`agreed===true`); when `agreed===false` the node is `geom-disagreement` carrying the
direction (Playwright's authoritative verdict + reason, and what geometry read), never a
code that contradicts the verdict. Within the agreed branch, PLAYWRIGHT'S NAMED CAUSE WINS
over geometry's: verdict-agreement is not cause-agreement (a disabled AND covered button has
both engines saying NOT-actionable, but Playwright's cause is `disabled`, not `covered`), so
we prefer the cause Playwright's error names (→ `confirmed`) and demote a geometry-only pick
Playwright did not corroborate to `suspected`. Diagnosis is surfaced only through an opt-in
serializer flag; the default serialized bytes are unchanged.

**Context:** `geometryVerdict` only detects geometry-visible blocks (off-screen, covered,
not-visible, pointer-events); disabled / read-only / animating are Playwright-only, so those
nodes have `agreed===false`. Emitting `disabled` as if geometry saw it would fabricate an
agreement that isn't there and risk contradicting the verdict on the inverse case (a
fillable covered input Playwright deems ACTIONABLE). Agree-or-flag keeps every confident code
backed by two engines and demotes single-engine hunches to a flagged disagreement — the
conservative posture that protects DW-02 and defers to the measured harness for the rest.

**Alternatives rejected:**
- Map Playwright's reason string straight to `disabled`/`read-only`/etc. regardless of
  agreement — over-claims `confirmed` from one engine and reintroduces the drift DW-02 exists
  to prevent; those causes still surface, in the geom-disagreement detail.
- Let diagnosis add or drop nodes (a membership filter) — geometry must never change the
  delta's membership (existing decision); diagnosis annotates, never filters.
- Always-on diagnostics in the serialized output — would move the default `@playwright/test`
  bytes and the token budget; the flag keeps the default path unchanged.

**Affected areas:** `src/host/diagnose.ts`, `src/host/serialize.ts` (opt-in flag),
`src/host/types.ts` (`Diagnosis`/`DiagnosedDelta`), `src/index.ts`, `test/diagnose.spec.ts`.

**Tags:** `accuracy` `diagnosis` `contract`

## 2026-07-08 — Gap-E late wave: detect with a separate observer, never capture (flag-not-fix)

**Decision:** The opt-in `lateWatchMs` closes silent gap E (a two-wave render whose second
wave lands after settle). When `lateWatchMs > 0`, `waitForSettle` still resolves AT the settle
point (unchanged), so the host's `collect()` runs there and the delta is frozen exactly as the
default path. At that settle point it also starts a SEPARATE, short-lived MutationObserver that
only flips a boolean on a structural mutation, bounded to the window. After collect + reconcile
the host calls `lateResult()`, which waits out the remaining window and reports `lateStructural`;
`diagnose` maps that to `late-wave-suspected` (suspected). Default `lateWatchMs=0` keeps the
settle path and delta byte-identical, and the `lateStructural` stats field is omitted entirely.

**Context:** A >quietMs inter-wave gap made Deltawright confidently-wrong on the highest-volume
flake mode; the gap must move from SILENT to FLAGGED before the harness numbers (#52) mean
anything. But CAPTURING and merging wave 2 into the delta was declined-as-unsafe in #30 (the
in-window signals that would gate it also false-drop legitimately-streamed payloads). So the
flag must observe the late wave WITHOUT letting it into the delta.

**Correction (adversarial review, M1):** an earlier revision STOPPED the main observer at the
settle point and delayed `waitForSettle`'s resolution by `lateWatchMs`, which deferred `collect`
to `settle + lateWatchMs`. Because `collect`/`coalesce` read the LIVE DOM (isConnected, geometry,
elementFromPoint), a *replacing* wave 2 (one that removes wave 1) would then ERASE wave 1 from
the delta — the opposite of "frozen default + flag". The fix keeps collect at the settle point
and moves the watch to a post-collect `lateResult()`; a `#replace` fixture pins it.

**Alternatives rejected:**
- Stop the main observer at settle and delay `collect` by the window — records are frozen, but
  `collect`'s LIVE reads run late, so a replacing/overlapping wave 2 alters or erases the delta
  (the M1 bug above).
- Keep the main observer running and read `lastStructuralAt > settlePoint` — the main observer
  would record wave 2 into `records`, so `collect` would CAPTURE it (declined-as-unsafe, #30).
- Capture wave 2 as a merged second delta — the #30 decision; unsafe.

**Known limitation:** the late observer watches the light DOM only; a late wave inside an
existing open shadow root is not detected (an accepted false-negative for a suspected hint).

**Affected areas:** `src/injected/observer.ts` (`waitForSettle`, `startLateWatch`, `lateResult`,
`lateObserver`, `reset`), `src/host/types.ts` (`SettleOptions.lateWatchMs`, `DeltawrightApi.lateResult`,
`DeltaStats.lateStructural`), `src/host/actAndObserve.ts` (opt-in wiring, conditional stats field),
`src/host/diagnose.ts` (`late-wave-suspected`), `test/late-wave.spec.ts` + `test/fixtures/late-wave.html`.

**Tags:** `accuracy` `settle` `design-panel` `contract`

## 2026-07-08 — Gap-F stale rect: re-read and adopt the later rect; annotation only

**Decision:** The opt-in `rectRecheckMs` closes silent gap F (a post-settle JS reposition
leaving a stale annotated rect). When `rectRecheckMs > 0`, AFTER the authoritative Playwright
probe, the host calls `recheckRects(ms)` — which waits, then re-reads every stamped node's current
geometry — and on a >2px move ADOPTS the later read, sets `GeometryRead.stable=false`, and
re-derives the geometry annotation (`geometryVerdict`/`agreed`). `diagnose` maps that to
`stale-rect-suspected` (suspected). Default `rectRecheckMs=0` leaves the annotated rect, the
(absent) `stable` field, the annotation, and stats byte-unchanged.

**Context:** `getAnimations()` is empty for a plain JS style write, so `settleAnimations` cannot
wait out a JS-timer recenter that fires after settle; the rect read at settle is then stale. A
side-car that shipped that stale geometry as fact would be confidently-wrong. Re-reading fixes
the annotation (adopts the accurate later rect) AND flags that it was stale.

**Why this can't touch the verdict — and the correction that made it true (adversarial review):**
geometry is ANNOTATION; Playwright's verdict is produced by the host probing the LIVE element
(via `data-dw-ref`). The FIRST revision ran the re-read *inside* `collect`, which the host awaits
BEFORE the probe — so enabling the flag delayed the probe by `rectRecheckMs`, and on a
still-evolving page (e.g. a node that moves off-screen in the window) the probe could see a
different state and return a different verdict. The flag then changed the verdict via TIMING, not
via the annotation. The fix moves the re-read to `recheckRects()` called AFTER the probe, so the
verdict is decided at the settle point regardless of `rectRecheckMs`; the strengthened test moves
the node off-screen and asserts the verdict stays `ACTIONABLE` while the geometry annotation flips.

**Alternatives rejected:**
- Re-read inside `collect` (the first revision) — defers the Playwright probe by `rectRecheckMs`,
  so the flag can change the verdict via probe timing (the review's finding above).
- Wait out the recenter before reading (extend `animMaxMs`) — `getAnimations` is empty, so there
  is nothing to await; only a timed re-read catches a JS write.
- Leave the stale rect and flag only — loses the chance to ship the ACCURATE rect for free;
  adopting the later read is strictly better and never risks the verdict.
- Always re-read — adds a `rectRecheckMs` delay and a `stable` field to every default delta;
  opt-in keeps the default byte-unchanged.

**Known limitation:** `recheckRects` re-reads via `querySelectorAll('[data-dw-ref]')` (light DOM
only), so a node inside a shadow root is not rechecked. Viewport-relative rects mean a post-settle
page scroll legitimately makes them stale (the flag + adopted rect correctly reflect the new
viewport position).

**Affected areas:** `src/injected/observer.ts` (`recheckRects`), `src/host/types.ts`
(`SettleOptions.rectRecheckMs`, `GeometryRead.stable`, `DeltawrightApi.recheckRects`),
`src/host/actAndObserve.ts` (post-probe re-read + `geometryVerdict` re-derive), `src/host/diagnose.ts`
(`stale-rect-suspected`), `test/stale-rect.spec.ts` + `test/fixtures/stale-rect.html`.

**Tags:** `accuracy` `geometry` `design-panel` `contract`

## 2026-07-08 — A non-circular flake corpus: three oracles + a mandatory confuser per code

**Decision:** The accuracy ground truth (`bench/flake-corpus/`) is labeled from THREE independent
oracles — the real Playwright verdict (live run), a hand-authored construction manifest
(`code` + `confidence`), and `window.__truth` for hidden causes — and is NEVER scored against a
stored Deltawright output (the `CorpusCase` schema has no field for one; `load.ts` guards it).
Every taxonomy code carries a positive case AND a MANDATORY near-miss confuser. Cases are `live`
(real-DOM fixtures) where reproducible, `delta` (hand-built, honesty-stamped) otherwise. All
numbers are corpus-relative; real-production precision stays blocked on #25/#41.

**Context:** A confuser-free, one-fixture-per-code corpus is a strawman — the author picks both
the DOM and the label, so precision is meaningless. The confuser (a DOM that resembles code X but
is NOT X) is what forces the engine to discriminate. Scoring against a saved diagnose output would
be circular (it would just prove the engine equals itself).

**What the corpus immediately surfaced (a live probe, recorded in CORPUS.md):** the engine emits
~9 of 18 codes well from real pages, and has honest gaps — `pointer-events-none` is mislabeled
`covered-by-overlay` (Playwright's error is a generic "intercept"); `disabled`/`read-only`/
`unstable-animating` surface as `geom-disagreement` (geometry can't see them); `detached-re-render`,
`injection-blocked`, `cross-boundary-partial`, and `pixel-region-fallback` aren't emitted yet
(missing signals / unmapped node). These are exactly why diagnosis capabilities are gated behind
the harness floor (#52) — the corpus makes the gaps measurable before anything ships on them.

**Alternatives rejected:**
- Score against saved `diagnose()` outputs — circular; proves nothing.
- One fixture per code, no confusers — inflates precision; the plan's named strawman.
- Only reproduce codes the engine already emits well — hides the gaps the corpus exists to reveal.

**Affected areas:** `bench/flake-corpus/{cases.ts,load.ts,CORPUS.md,fixtures/reveal.html}`,
`test/corpus.spec.ts`.

**Tags:** `accuracy` `corpus` `contract`

## 2026-07-08 — diagnose(): geometry's self-cause beats Playwright's generic "intercept" (#71)

**Decision (extends the #48 agree-or-flag / PW-cause-wins rule):** Playwright's "intercepts
pointer events" error is generic — it fires for a real overlay AND when the target's own
`pointer-events:none` makes it not receive the hit (where `elementFromPoint` returns the element
BEHIND, so `coveredBy` is set to an incidental node). When the target's own computed
`pointer-events` is `none`, `diagnose()` now names `pointer-events-none` (suspected — a
geometry-only specific cause Playwright confirmed only as "blocked"), NOT `covered-by-overlay`.
Separately, the screenshot-diff pixel-region node is now mapped to `pixel-region-fallback`.

**Context:** The #51 corpus + a live probe showed the engine emitting a WRONG `confirmed`
(`covered-by-overlay`) for a `pointer-events:none` target — a confidently-wrong label, the
product's anti-goal, and a direct threat to #52's ≥0.95 confirmed-precision floor. The #48
rule ("Playwright's named cause wins") is right when Playwright names a cause geometry can't see
(disabled); it is WRONG when Playwright's error is a generic symptom and geometry has the precise
self-cause. So the refinement is narrow: only the `pointer-events:none` self-cause overrides the
generic intercept, and only to `suspected` (not over-claiming).

**Alternatives rejected:**
- Keep `covered-by-overlay` — a wrong `confirmed`; breaks the precision floor and mislabels.
- Reorder `codeFromGeometry` to put pointer-events before coveredBy — would also mis-order the
  human reason string vs the code for the covered case; the explicit narrow branch is safer.
- Emit `pointer-events-none` as `confirmed` — Playwright disagreed on the specific cause, so
  `suspected` is the honest band.

Remaining #71 gaps (disabled/read-only recall via the disagreed branch; detached/injection/
cross-boundary silent misses) need new signals or an agree-or-flag ADR and stay open.

**Affected areas:** `src/host/diagnose.ts` (`blockingDiagnosis`, pixel-region mapping),
`test/diagnose.spec.ts`, `bench/flake-corpus/{cases.ts,CORPUS.md}`.

**Tags:** `accuracy` `diagnosis`

## 2026-07-10 — diagnose(): recover geometry-blind causes from the disagreed branch (#71)

**Decision (extends the #48 agree-or-flag rule):** The three causes only Playwright can observe —
`disabled`, `read-only`, `unstable-animating` — are now emitted from the DISAGREED branch
(`agreed===false`) as `confirmed`, instead of collapsing to `geom-disagreement`. When a
NOT-actionable node reads geometry-ACTIONABLE and Playwright's error names one of these
GEOMETRY-BLIND causes, `diagnose()` recovers that Playwright-named cause. The recovery is strictly
limited to a closed blind set (`GEOMETRY_BLIND_CAUSES = {disabled, read-only, unstable-animating}`);
a dissent on any geometry-VISIBLE cause (covered / off-screen / not-visible / pointer-events) is
genuine counter-evidence and still yields `geom-disagreement`.

**Context:** `geometryVerdict` reads rect / style / `elementFromPoint` only — it has NO signal for
the enabled, editable, or stability state. So a disabled button, a read-only input, and a
mid-animation element all read geometry-actionable and land in the disagreed branch, where #48
routed them to `geom-disagreement`. That was a RECALL gap the #51 corpus surfaced: the true root
cause (which Playwright authoritatively named in its error) was demoted to "the engines disagree."
The key insight distinguishing this from #48's conservative default: geometry's "actionable" read
on a blind cause is the ABSENCE of evidence about that cause, **not evidence against it**. It is
therefore not a real conflict, so recovering the Playwright-named cause neither fabricates a
false agreement nor contradicts the verdict — the diagnosis (`disabled`) *is* the verdict's
(NOT-actionable) reason. Confidence is `confirmed` for the same reason it is in the agreed branch:
an authoritative engine named the cause (`assessConfidence({source:'playwright'})`); the
`conflicting` downgrade does NOT apply, because geometry casts no competing vote on a cause it
cannot see. Geometry's read is preserved in the diagnosis detail for transparency.

**Why the blind-set gate is load-bearing (the review's crux):** without it, "Playwright named a
cause → recover it" would also fire when Playwright reports a generic intercept
(`covered-by-overlay`) but geometry sees nothing covering the target — a case where geometry IS
competent and its dissent is real counter-evidence. Emitting `covered-by-overlay` confirmed there
would over-claim against a competent disagreement (and is the inverse of the #71 pointer-events
fix). Partitioning the taxonomy into geometry-blind vs geometry-visible causes is what keeps the
recovery sound: recover only where geometry is provably blind; flag everywhere geometry can see.

**Alternatives rejected:**
- Recover any Playwright-named cause from the disagreed branch (drop the blind-set gate) —
  over-claims `covered-by-overlay`/`off-screen` confirmed on a competent geometry dissent, the
  exact drift DW-02 exists to prevent.
- Emit the recovered cause as `suspected` to hedge — under-sells a cause Playwright
  authoritatively named and is inconsistent with the agreed branch (where the same PW-named cause
  is `confirmed`); geometry blindness is not a reason to doubt Playwright.
- Also emit a companion `geom-disagreement` alongside the recovered cause — noise: the
  disagreement is fully explained by "geometry is blind to this cause," and the detail already
  carries geometry's read.

**Affected areas:** `src/host/diagnose.ts` (`GEOMETRY_BLIND_CAUSES`, `blindCauseDiagnosis`,
`diagnoseNode` routing), `test/diagnose.spec.ts`, `bench/flake-corpus/{cases.ts,CORPUS.md,
fixtures/reveal.html}` (a live `covered-input` reveal replaces `disabled` as the genuine

ACTIONABLE node) and DW-03 (confidence stays engine-owned).

## 2026-07-10 — diagnose(): detached-re-render from an in-window detach signal (#71 fix #3)

**Decision:** A freshly-added subtree that is inserted and then DETACHED again within the settle
window (a React re-render / list-virtualization swap) now grounds `detached-re-render`. `coalesce`
counts elements first-touched as an ADD that end the window disconnected (reduced to subtree roots),
sets a default-absent `stats.detachedReRender` when the count is > 0, and `diagnose()` maps that to
a delta-level `detached-re-render` diagnosis at **suspected** confidence. Extends the #48
diagnosis engine; does not touch any node verdict (DW-02).

**Context:** The scenario is real: an action produces a node, then a framework re-render tears it
down and replaces it, so a handle/ref an agent captured is stale. But such a node nets OUT of the
reported delta — it is excluded from `addedRoots` (no longer `isConnected`) AND from `removed`
(`firstTouch !== 'remove'`, it was born this window) — so the delta shows only the replacement,
cleanly actionable, and the transience was previously invisible (a silent miss in the #52 harness).
The information is already computed inside `coalesce` (it walks the added candidate set to build
`addedRoots`); counting the detached ones is zero added latency and needs no new observer.

**Why SUSPECTED, not confirmed:** an add-then-detach is not proof of a re-render — a benign
transient (a loading spinner that appears then vanishes, replaced by content) produces the same
observer signature. The signal honestly says "a freshly-added node was detached mid-window; a handle
to it would be stale," and the confidence band reflects that it is an attribution heuristic, not an
authoritative engine verdict. It is delta-level (not attached to any surviving node) precisely
because the detached node is gone; the replacement's own verdict is untouched.

**Honors the background quarantine (review fix).** The count applies the SAME `bgInsert`
background-insertion exclusion the connected added-root path uses, so a RECURRING background
insertion learned in the pre-arm baseline (an auto-dismissing toast, a virtualized row, a spinner
that cycles) — the exact churn the delta already drops — cannot trip `detached-re-render`. This
needed care: a detached node's `parentElement` is `null` by collect, so the node's signature is
snapshotted at ADD time from the mutation record's parent (`rec.target`), not recomputed at collect.
Without it, a churny page would co-fire a spurious "a handle is stale" note next to `background-churn`,
contradicting the delta's own background filtering. A live regression test pairs a recurring
background toast with a persistent modal and asserts the flag stays absent (and, with baseline off,
that it does fire — proving the guard is not vacuous).

**Scope (honest bounds).** This detects the IN-WINDOW add-then-detach sub-case only. A keyed-list
reorder (a move nets to nothing structurally), a detach inside a shadow root or child frame (the
`stats` object is main-frame-scoped, like every other `DeltaStats` field), and a re-render that
detaches a stamped node AFTER collect are all out of scope for this signal; the post-collect
ref-staleness window remains available as future hardening (see below).

**Why observer-level (add-then-detach), not reconcile-level (ref-staleness):** the alternative was
to flag a node whose stamped `data-dw-ref` fails to resolve during reconciliation. That only catches
a detach in the narrow collect→probe window (milliseconds) and would need the fixture re-rendered
to detach AFTER stamping. The observer-level count catches the far more common case — a re-render
DURING settle — with the existing microtask fixture and no extra probe. Reconcile-level ref-staleness
remains available as future hardening for the post-collect window.

**Alternatives rejected:**
- Report the detached node itself as a `removed` node — it was never a net removal (it was born and
  died within the window); surfacing it as removed would misrepresent the DOM's net change and
  pollute the delta. The delta stays the net truth; the detach is a separate diagnostic note.
- Gate the count behind an opt-in flag (like gap-E `lateWatchMs` / gap-F `rectRecheckMs`) — those
  flags exist because they add post-settle latency; this count adds none, so it is always-on with a
  default-ABSENT field, keeping the default `stats` object byte-unchanged when no detach occurred.
- `confirmed` confidence — would over-claim on benign transients; violates DW-03 (only an
  authoritative engine earns confirmed).

**Affected areas:** `src/injected/observer.ts` (`coalesce`, `collect`, `NetDelta`),
`src/host/types.ts` (`CollectResult.detachedInWindow`, `DeltaStats.detachedReRender?`),
`src/host/actAndObserve.ts` (default-absent mapping), `src/host/diagnose.ts` (`diagnoseDelta`),
`bench/flake-corpus/cases.ts` + `CORPUS.md` (the case is now a hit), `test/diagnose.spec.ts` +

## 2026-07-10 — diagnose(): capture-integrity codes injection-blocked + cross-boundary-partial (#71 fix #4)

**Decision:** Two capture-integrity signals close the last two #71 silent misses, both plumbed from
the primitive into `DeltaStats` and mapped by `diagnose()`:

- **`injection-blocked` (confirmed).** When the observer cannot be injected — `addScriptTag` throws,
  typically under a strict CSP (`script-src 'none'`) — `actAndObserve` no longer throws out of the
  primitive. It DEGRADES: catches the failure, still performs the action (act-and-observe must ACT),
  and returns an empty delta carrying `stats.injectionBlocked=true`. `diagnose()` maps that to
  `injection-blocked` at **confirmed** confidence — the injection failure was authoritatively
  observed (the throw), not inferred. To keep that `confirmed` honest (review finding), only an
  authoritative `addScriptTag` rejection degrades: `ensureInjected` raises a typed
  `InjectionBlockedError` from the `addScriptTag` catch alone, and the degrade path re-throws
  everything else — a pre-inject presence-probe throw, a bundle-load failure, or a transient
  navigation-race error stays loud and retryable rather than being mislabeled as a confirmed CSP
  block (DW-03).
- **`cross-boundary-partial` (suspected).** During `frames:true` traversal, `armChildFrames` already
  catch-skips cross-origin / uninjectable child frames; it now COUNTS those skips (plus any frame
  that detaches during the collect loop) into `stats.crossBoundarySkipped`. `diagnose()` maps a
  non-zero count to `cross-boundary-partial` at **suspected** — we know a boundary was skipped, but
  not whether the action changed anything behind it.

**Context:** These were the two remaining `diagnose()` silent misses in the #52 harness. Both are
capture-integrity failures — the delta is empty or partial NOT because nothing changed but because
something could not be observed — and presenting that as a confident no-op would be exactly the
"confidently wrong" the product's anti-goal forbids.

**Why `injection-blocked` still runs the action:** the alternative — return the flag WITHOUT acting
— would silently drop the caller's intended side-effect (the click/fill they asked for), which is
more surprising than degrading the observation. The primitive is *act*-and-observe; if observe is
impossible, act still happens and the flag honestly reports that the effect couldn't be captured.

**Why closed shadow roots are not counted:** the plan sketched "count skipped closed shadow roots,"
but a closed shadow root is structurally undetectable from script — `el.shadowRoot` returns `null`
for both "no shadow" and "closed shadow" — so there is nothing reliable to count. The honest,
countable capture gap is skipped *frames*; the doc was corrected to say so.

**Corpus:** BOTH are promoted to **live** cases (review finding — a hand-authored stat left the real
skip-counting path untested). `injection-blocked` runs `fixtures/csp-blocked.html` (a
`<meta CSP script-src 'none'>` page). `cross-boundary-partial` runs `fixtures/cross-boundary.html`,
whose child frame carries a srcdoc CSP that blocks its injection, so the real `armChildFrames` skip
path increments `crossBoundarySkipped` — a same-origin CSP-uninjectable child stands in for the
cross-origin case (a true cross-origin child needs two origins → #25). No hand-authored
capture-integrity stat remains.

**Alternatives rejected:**
- Throw on injection failure (status quo) — the caller gets an exception, not a diagnosable delta;
  hides a recoverable, nameable failure mode behind a stack trace.
- `injection-blocked` at suspected — under-claims; we directly observed the injection throw.
- `cross-boundary-partial` at confirmed — over-claims; a skipped frame is a capture gap, not proof
  the action changed something inside it.
- Detect closed shadow roots by hooking `Element.prototype.attachShadow` — requires injecting before
  page scripts run (which is the very thing blocked here) and is far more invasive than the value of
  a suspected hint; declined.

**Affected areas:** `src/host/actAndObserve.ts` (degraded injection path + `armChildFrames` skip
count + stats mapping), `src/host/types.ts` (`DeltaStats.injectionBlocked?` / `crossBoundarySkipped?`),
`src/host/diagnose.ts` (`diagnoseDelta`), `bench/flake-corpus/cases.ts` + `fixtures/csp-blocked.html`

## 2026-07-10 — Delta checksum matcher: regression-only, geometry-tolerant, structural diff (#54)

**Decision:** `deltawright/matchers` adds `expect(delta).toMatchDeltaChecksum(id)` and
`expect(delta).toMatchDeltaSnapshot()` over the existing `checksum` / `normalizeDelta` primitive. The
baseline is the canonical normalized form (geometry/timing-tolerant), stored one file per id in
`__dw_checksums__/` next to the running spec; it is written on first run and refreshed only in an
explicit update mode (`DW_UPDATE_CHECKSUMS=1`, Playwright `--update-snapshots`, or
`deltawright checksum --update -- <cmd>`). A mismatch renders a STRUCTURAL (LCS line) diff of the
pretty-printed baseline vs current, never a pixel diff. Every message carries the regression-only
honesty note.

**Context:** For a large suite this is the cheapest, engine-independent guard: it pins that an action's
delta hasn't structurally changed while tolerating the pixel/timing/record-order jitter that makes raw
snapshots flaky (`checksum.ts` already normalizes those out). It packages a shipped primitive; no new
accuracy surface.

**Honesty is load-bearing (per checksum.ts's own note):** a green checksum proves output ==
previously-captured output — a REGRESSION guard and nothing more. It says nothing about whether the
fixture models a real app or that Deltawright "works". The matcher refuses to let that be forgotten:
the note is emitted on write, on match, AND on mismatch, and a test asserts it. A known **blind spot**
carried from the fingerprint (`checksum.ts`) is stated too: `changedAttrs` is not hashed, so the
checksum records THAT a node's attributes changed, not WHICH ones — an attr-identity swap with an
unchanged verdict/tree is not distinguished (folding the attr set back into the fingerprint is a
candidate refinement, deferred because it changes the shipped `checksum()` output + a frozen guard).

**Design choices (with the review fixes folded in):**
- Store the canonical normalized form (not just the sha256) so a mismatch can render a structural
  diff — the hash alone can't. Matching is still byte-equality of the geometry-tolerant fingerprint.
- A pure `matchDeltaChecksum(delta, file, {update, failOnMissing})` core does all the work (filesystem
  injected), so the matcher is a browser-free unit; the thin `expect` wrapper only resolves the
  baseline path + modes from `test.info()` + env (baselines next to the spec).
- MISSING-baseline semantics (review fix): a local first run writes + passes (jest-style), but in **CI**
  (`process.env.CI`, threaded as `failOnMissing`) a missing baseline is written yet FAILS — so an
  uncommitted / gitignored / relocated baseline can't first-run-green forever and quietly guard
  nothing. An explicit update always passes.
- Update fires on `DW_UPDATE_CHECKSUMS=1` OR Playwright's `updateSnapshots` being `'all'` **or**
  `'changed'` (review fix — the idiomatic bare `-u` / `--update-snapshots` resolves to `'changed'` in
  Playwright 1.61, so honoring only `'all'` broke the documented native refresh path).

**Alternatives rejected:**
- A pixel/image diff — exactly the flaky, geometry-coupled comparison the delta fingerprint exists to
  avoid; the diff must be over the normalized structure.
- Store only the checksum hash — compact, but a mismatch could then only say "hash A != hash B" with
  no structural diff; the acceptance criteria require the diff.
- A bespoke `DW_UPDATE` env as the ONLY update path — also honor Playwright's `--update-snapshots`
  (`-u`) so it composes with a normal update workflow; the CLI and env are conveniences on top.
- Green-on-missing everywhere (the naive snapshot convention) — safe locally but lets CI green on an
  unestablished baseline; guarded via `failOnMissing` in CI.

**Affected areas:** `src/matchers/checksum.ts` (pure core + structural diff), `src/matchers/index.ts`
(`toMatchDeltaChecksum` / `toMatchDeltaSnapshot` wrappers + combined `dwMatchers` bag + augmentation),
`src/cli.ts` (`checksum --update`), `test/checksum-matcher.spec.ts`, `test/packaging.spec.ts` + the
consumer fixture. Regression-only; real-app fidelity stays a separate, u/nclaimed question (#25/#41).

**Tags:** `matcher` `regression` `contract`

(`*.deltawright-sidecar.json` + `*.triage.txt`) for every FAILED / TIMED-OUT test — never on a passing
test, never altering pass/fail (a reporter structurally cannot). It CONSUMES the one `diagnose()`
engine, so it hard-gates on the #52 accuracy harness. All logic is a pure, browser-free
`triageFailure(input) → Sidecar | null` core; the Reporter is a thin wrapper (map TestCase/TestResult
→ input, write files).

**Two modes, one engine:**
- PASSIVE (zero-edit): the failing action's Playwright actionability error is wrapped in a SYNTHETIC
  one-node delta and run through `diagnose()`. Playwright's own failure error is an authoritative
  actionability signal (the same source `diagnose()` treats as authoritative from a trial probe), so
  this yields a real taxonomy code + confidence with no test changes and no browser.
- RICH (opt-in, a test edit): a test attaches a real delta via `attachDelta(testInfo, delta)`; the
  reporter diagnoses THAT, so the side-car carries geometry-grounded causes plus the late-wave /
  stale-rect flags (present only when the delta was captured with `lateWatchMs` / `rectRecheckMs`).
  The "zero-edit" claim is scoped to the passive path only.

**Never fabricates (honesty — hardened by the review):** the passive path is GATED on the error being
a genuine Playwright actionability failure (`isActionabilityError`: the structured `element is not
visible|enabled|stable|editable|attached` / `intercepts pointer events` / detach phrasings), NOT a
bare keyword anywhere in the text. Without the gate, an assertion diff or stack trace that merely
CONTAINS a taxonomy word ("disabled", "animate", "hidden") would be substring-matched into a confident
wrong cause — the review reproduced exactly that. Unrelated failures now stay `unsure`. A cause below
the confidence threshold (default `suspected`) is `unsure`; a locator that never resolved — "not
attached to the DOM", "was detached from the DOM" (the auto-waiting retry phrasing), "no element
matching", "target … has been closed" — degrades to `detached` + `unsure`, short-circuiting the
diagnosis so a later incidental keyword in the same call log can't be mined into a cause. A malformed
`deltawright-delta` attachment (validated node shape) falls back to passive triage rather than
throwing, and the whole triage computation is guarded so it can never break the run. Finally, only a
test whose FINAL outcome is a real failure gets a side-car (flushed in `onEnd` via `test.outcome()`),
so a flaky-then-passed test leaves no stale failure record. The triage text stamps that a cause is a
HYPOTHESIS from Playwright's failure signal, not proof.

**Alternatives rejected:**
- Live-probe the failing page from the reporter — impossible: `onTestEnd` runs in the main process
  after the worker's browser is gone. The passive synthetic-delta route is the honest zero-edit path;
  live geometry needs the opt-in in-worker attachment.
- A parallel error→code mapping in the reporter — would drift from `diagnose()`; routing the synthetic
  delta THROUGH `diagnose()` is what makes the reporter hard-gate on the harness (no second engine).
- Emit a best-guess cause for a detached/timed-out locator — that is exactly the confidently-wrong the
  product forbids; degrade to `unsure`/`detached` instead.

**Affected areas:** `src/reporter/triage.ts` (pure core + render), `src/reporter/index.ts` (Reporter +
`attachDelta` helper + subpath surface), `package.json` exports + `scripts/build.mjs` (`./reporter`),
`test/reporter.spec.ts` (5 criteria), `test/packaging.spec.ts` + the consumer fixture. Rich-mode late-
wave/stale-rect need the capture flags on the producing `actAndObserve`.

**Tags:** `reporter` `diagnosis` `contract`

## 2026-07-10 — suggest(delta): pure test-authoring aid, honest about what it can't ground (#57)

**Decision:** `suggest(delta): { assertions, selectors, warnings }` is a PURE function exported from the
main entry (`src/host/suggest.ts`, like `diagnose` — not a matcher, not a new subpath). It proposes
candidate selectors for each changed node ranked **getByRole > getByText > testid > css**, and a
`toBeActionable()` assertion ONLY for nodes whose actionability verdict is `ACTIONABLE`. It reads only
the delta — no Playwright, no I/O, no input mutation — and never writes or modifies a test file. First
of the Wave-2 capabilities, unblocked because the accuracy harness held its gated floors through all of
Wave-1.

**Honest by construction (the load-bearing part):** every tier is grounded in a real `DeltaNode` field
or explicitly withheld:
- `role` → `getByRole(role, { name })`, `name` → `getByText(name)`, `css` → `tag` (+ optional `[role]`).
- **`testid` is NEVER emitted** — the delta carries no test-id attribute, so a `getByTestId` would be
  fabricated; it is withheld and warned. The rank keeps `testid` above `css` so it slots correctly if
  a future observer captures the configured `testIdAttribute`.
- **`data-dw-ref` is NEVER offered** as a selector — it is stripped on the next action (ephemeral).
- Warnings state that role/name are Deltawright's HEURISTIC reads (not Playwright's accessible-name/role
  algorithm), that a `name` may be an aria-label (so `getByText` can miss), and that `suggest()` sees
  one delta in isolation so it does NOT verify uniqueness or stability. The **durable-selector problem
  is explicitly a separate design investigation**, not delivered here (per #57).

**Why it consumes the one engine, not a second mapping:** it reuses the actionability verdict already
on the delta to gate `toBeActionable()`, and the role/name the observer already recorded — introducing
no parallel role/name logic that could drift from the delta or the matcher.

**Alternatives rejected:**
- Emit a `getByTestId` guess or a `data-dw-ref` selector — both would be confidently-wrong (fabricated
  or ephemeral); withhold + warn instead.
- Solve durable/unique selectors here — genuinely unsolved from a single delta (no page-wide view);
  scoped out to a separate investigation, and warned rather than silently over-promised.
- Put it behind `deltawright/matchers` — that subpath imports `@playwright/test` and writes files; a
  pure delta→suggestions function belongs on the browser-free main entry next to `diagnose`.

**Selector-code validity (review fixes):** the emitted code must be VALID pasteable Playwright. Two
execution-reproduced bugs were fixed: (1) `q()` now escapes `\n`/`\r` (an aria-label / placeholder
keeps internal newlines, which would otherwise make a single-quoted literal a syntax error); (2) the
css tier is the **bare tag**, not `tag[role="…"]` — the delta's `role` is the observer's RESOLVED
(often implicit) role, so a native `<button>` (role `button`, no `role=` attr) would get
`button[role="button"]` which matches nothing; the honest coarse fallback is the tag alone (this also
removes an unescaped-role injection into the CSS).

**Affected areas:** `src/host/suggest.ts` (new), `src/index.ts` (export), `test/suggest.spec.ts`,
`test/packaging.spec.ts` + the consumer fixture. Capturing a real test-id / a11y name, and the
durable-selector investigation, are follow-ups.

**Tags:** `test-gen` `contract`

## 2026-07-10 — observeConsequences: settle-as-a-wait, a signal NOT a guarantee (#58)

**Decision:** `deltawright/wait` ships `observeConsequences(page, action): { settleMs, hitMaxWait,
suspectedEarly, observed, skippedReason? }` — a locator-free completion SIGNAL. It arms the observer,
runs the action, waits for STRUCTURAL quiescence via the existing `waitForSettle`, reports the gap-E
(#49) late-wave heuristic as `suspectedEarly`, and SKIPS `collect`/reconcile (no per-node Playwright
probes, no geometry) — the O(nodes) reconcile is the real cost it saves vs `actAndObserve`. (Review
note: it is NOT unconditionally faster wall-clock — the default late-watch adds a fixed window, so on
a trivial single-wave page it can be slower; `lateWatchMs: 0` skips it. The cost claim is scoped to
the reconcile it avoids, and the docs say so.) It also drops a needless `pngjs` eager import by
sourcing `DEFAULT_SETTLE` from the side-effect-free `types` leaf, and guards observer teardown in a
`finally` so a throw mid-wait can't leak a running MutationObserver. `suspectedEarly` is disclosed as
a coarse hint (light-DOM only — misses open-shadow-root late waves — and unfiltered for background
churn). Second gated Wave-2 capability (the harness held its floors through Wave-1).

**NOT a guarantee — enforced in the type, not just docs:** the product's named non-goal is that
Deltawright does not fix/retry/suppress flakes. A "settle-as-a-wait" is the wrapper most easily
misread as `waitUntilReady()`. So the API is shaped to make that misreading impossible: the result
carries only settle SIGNALS — `settleMs`, `hitMaxWait` (settle was inconclusive — hit the cap), and
`suspectedEarly` (a late wave landed, so settle may have been early) — and NO `ready`/`safe`/`settled`
boolean and NO retry knob. A test asserts the exact key set. `hitMaxWait`/`suspectedEarly` are the
honesty signals the caller interprets; `observeConsequences` never claims the page is ready and never
silences a flake.

**Reuse, not a fork:** it uses the same injected `arm` / `waitForSettle` / `lateResult` the delta path
uses (no second settle heuristic), and `lateWatchMs` is ON by default here (unlike `actAndObserve`,
where it is opt-in) because the late-wave signal is the point. It disconnects the observer via `reset`
since it skips the `collect` that normally stops it. Under a strict CSP / non-Chromium it degrades
(only on `InjectionBlockedError`, DW-03): it still performs the action and returns `observed: false`
with a reason.

**Alternatives rejected:**
- A `waitForReady()` / boolean "settled" return — reads as a guarantee; the whole point is that a
  quiescence heuristic is a SIGNAL, and `hitMaxWait`/`suspectedEarly` exist precisely because it can be
  wrong. Expose the signals, not a verdict.
- A built-in retry/auto-wait-until-stable — that IS the flake-suppressant non-goal; declined.
- Route it through `actAndObserve` and ignore the delta — wasteful (it would run the O(nodes)
  reconcile); the value here is the cheaper reconcile-free settle.

**Affected areas:** `src/wait/index.ts` (new), `package.json` exports + `scripts/build.mjs`
(`./wait`), `test/wait.spec.ts` (3 criteria + CSP degrade + the anti-guarantee shape),
`test/packaging.spec.ts` + the consumer fixture.

**Tags:** `wait` `contract`

## 2026-07-11 — Checksum folds allowlisted changed-attribute NAMES (Wave-1 #3)

**Decision:** The delta checksum's canonical form now includes, per `attrChanged` node, the SET of
changed attribute NAMES filtered to a fixed **state allowlist** (`class`, `disabled`, `checked`,
`selected`, `open`, `hidden`, `readonly`, `contenteditable`, and `aria-pressed`/`-expanded`/
`-selected`/`-checked`/`-current`/`-disabled`/`-invalid`/`-hidden`/`-busy`/`-required`), lower-cased,
de-duped and **sorted** (`ca` field + `sortKey` fold). So a delta that starts toggling a *different*
state attribute (e.g. `class` → `aria-pressed`) now yields a different fingerprint even when the tree
and Playwright verdict are unchanged. The `GWT_APPROVE_CHECKSUM` lock was rebaselined once; the
frozen canonical form now carries `"ca":[…]` on every node.

**Context:** The prior fingerprint recorded THAT a node's attributes changed (`kind=attrChanged`) but
never WHICH — a named blind spot (`checksum.ts` header): an `aria-pressed`/`class` state-toggle swap
with an unchanged verdict/tree passed silently. `toMatchDeltaChecksum` is a regression guard, so
missing a real state-identity change is a real miss.

**Why an allowlist (not the raw attr set):** the module's governing rule is "drop anything that
jitters across runs." Only attribute NAMES survive capture (`observer.ts` discards values), and
background attr churn is already filtered upstream (`bgAttr`). The remaining jitter risks are (a)
record ORDER — handled by sorting; and (b) volatile attr NAMES (`style`, framework-generated tokens)
— handled by the allowlist, which admits only stable interaction/widget-state attrs. `class` is kept
by NAME only (that class changed), never by its churny token values.

This fulfils the "fold the attr SET back in" refinement deferred in the 2026-07-08 checksum-matcher
(#54) decision.

**Known limitation (honesty):** because values are not captured, the checksum distinguishes WHICH
state attribute changed, not its `old → new` value — it cannot tell `aria-pressed` true→false from
false→true. A non-allowlisted-only attr change (e.g. `style` alone) collapses to the empty attr set,
identical to no attr change. This is an accepted false-negative that keeps the coarse bucket coarse.
Because presence-of-a-changed-state-attr-at-settle now folds into the fingerprint, an app that ends
the settle window with a transient state attr still set (a lingering `aria-busy`, a `.loading` class)
contributes it — DETERMINISTIC per app (not run-to-run jitter: settle resolves on structural
quiescence, attr churn is net-compared, and the recurring-background set is dropped upstream), but it
means the fingerprint reflects the app's end-of-settle state, as intended for a regression guard.

**Alternatives rejected:**
- Fold the raw `changedAttrs` array — reintroduces record-order + volatile-name jitter the module
  explicitly drops.
- Capture attr VALUES to distinguish true↔false toggles — values are discarded at collect
  (`observer.ts`); adding them is a larger capture change out of scope and a fresh jitter surface.
- A denylist of volatile attrs — unbounded (can't enumerate every framework's generated names);
  an allowlist fails safe (an unknown attr is dropped, never hashed).

**Affected areas:** `src/host/checksum.ts` (`STATE_ATTR_ALLOWLIST`, `normAttrs`, `NormNode.ca`,
`sortKey`, `canonical`), `src/matchers/checksum.ts` (stale blind-spot comment corrected),
`test/checksum-matcher.spec.ts` (+2 sensitivity/jitter tests), `test/gwt.spec.ts` (rebaselined lock),
`dist/` (rebuilt so `packaging.spec` dist==src holds).

**Tags:** `checksum` `matchers` `contract` `regression-guard`

## 2026-07-11 — Composite GitHub Action: consume side-cars, gh-over-third-party, sticky, degrade-to-nothing (Wave-1 #4)

**Decision:** Ship a repo-ROOT `action.yml` composite action (`uses: mstomar698/deltawright@<ref>`)

**Context:** The reporter (#55), aggregator (#59), and dashboard (Wave-1 #2) exist as three subpaths;

**Why these choices:**
- **Consume the side-car dir, don't run the suite.** Running the user's tests would force their

- **Sticky via a hidden HTML-comment marker**, matching the repo's idempotent-step convention — the
  roadmap flagged comment noise as the risk.
- **Root `action.yml`** so `uses: owner/repo@ref` resolves; it is outside the npm `files` allowlist,
  so it never ships in the tarball (no package change → no version bump; rides the next release tag).

**Alternatives rejected:**
- A third-party sticky-comment action — smaller code but a supply-chain dependency the repo avoids.
- `gh pr comment` (appends) or `--edit-last` (clobbers any last comment) — the marker+PATCH path is
  precise and re-run-safe.
- Emitting the comment from a new structured `aggregate --json` — `--report` already carries the
  dominant category + rates per test and keeps the `unsure` bucket separate (DW-03); no new surface
  needed.

**Honesty/limits:** the comment carries the corpus-relative + Playwright-authoritative stamp and the
"unsure never folded into a category" note. The action can't be exercised on a real runner in-repo;
`test/action.spec.ts` pins the contract (composite, inputs, degrade guard, first-party-only, sticky
marker) and runs the two exact CLI commands it shells out to.

**Affected areas:** `action.yml` (new, root), `test/action.spec.ts` (new), `README.md` (CI-in-one-step
section), `docs/cookbook.md` (row). No `src/` or package change.

## 2026-07-11 — Durable-selector recommender: a page-aware verify layer over pure suggest() (Wave-2 #6)

**Decision:** Ship `verifySuggestions(root, delta)` on the impure `deltawright/matchers` subpath — a
PAGE-AWARE layer over the pure `suggest()` that #57 explicitly punted. For each candidate `suggest()`
proposes, it rebuilds the live Locator (role/name/tag → `getByRole`/`getByText`/`locator`) and resolves
it against the page: `locator.count()` for uniqueness, and same-element identity by intersecting the
candidate with `[data-dw-ref="<ref>"]` via Playwright's `.and()` combinator. Candidates are classified
`verified` (matches 1, IS the delta's element) · `ambiguous` (>1) · `unique-elsewhere` (1, wrong
element) · `broken` (0), re-ranked verified-first, and the best verified-unique one is surfaced as
`bestVerified`. `suggest()` stays PURE and byte-unchanged; this sits on top.

**Context:** `suggest()` sees one delta in isolation and cannot know if a candidate is unique or even
resolves — it warned about this and #57 scoped the durable-selector problem to a separate
investigation. #6 adds the page-wide view that closes it: turning "a candidate to hand-verify" into a
paste-ready, verified-unique locator (or an honest "nothing is unique").

**Honest limit on "stability":** the roadmap said "score stability across deltas," but there is NO
stored multi-delta selector history in the codebase (the aggregator reads diagnosis side-cars, not
selectors; checksum baselines carry no selectors). So true cross-RENDER stability is out of reach
without new plumbing and is NOT claimed. "Stability" here means single-page **uniqueness +
same-element identity**, which is fully page-authoritative. This mirrors #57's own honest framing
("genuinely unsolved from a single delta (no page-wide view)") — #6 supplies exactly the page-wide
view, no more.

**DW-03 (stay unsure):** if no candidate resolves uniquely to the changed element, `bestVerified` is
`null` and a warning says so — never hand back an ambiguous locator as if durable. A second warning
fires when the delta's `data-dw-ref` is absent (it is cleared on the next `arm()`), so same-element

- Edit `suggest()` to take a Page — breaks its purity contract + the enforcing test; the whole value

**Review-hardened (2 MEDIUM, both DW-03 honesty gaps, fixed before merge):**
- *Pass-through assertions* — `suggest()` builds each `toBeActionable()` on the node's TOP selector; the

  selector, and dropped (with a warning) when the node has none.
- *Match-gated ref-presence* — ref presence was inferred only when a candidate matched, so an
  all-broken node could fire a spurious "you called too late" warning, and a node whose ref was gone (a
  re-render after collect) had its unique match mislabeled `unique-elsewhere` ("wrong element") when
  identity was merely *unconfirmable*. Now ref presence is probed per node UP FRONT (independent of
  matching); the refs-absent warning is driven off actual presence; and a fifth status **`unconfirmed`**
  (matches 1, ref gone) keeps absence-of-evidence from reading as evidence-of-wrong.

## 2026-07-11 — Background-churn: detect post-action recurrence + flag SUSPECTED; do NOT change settle/attribution (Wave-2 #7)

**Decision:** Add an in-window insertion-recurrence signal that detects a container which STARTS
churning AFTER the action (a polling feed, a live-updating region) — invisible to the pre-arm
baseline — and surfaces it as `background-churn` (**suspected**). The observer tallies raw insertion
signatures per window (`winInsertCount`); at collect it reports the peak recurrence for a signature
NOT already in the pre-arm `bgInsert` baseline (`CollectResult.recurringInsert`). The host adds the
default-absent `DeltaStats.recurringInsert` ONLY when it crosses the threshold (`RECUR_MIN = 4`) AND
settle hit the cap (`hitMaxWait`) — churn that also kept settle from quiescing (see the honest
residual below). `diagnose()` grounds the existing
`background-churn` code on this (no new code → the frozen taxonomy SHA is untouched; the new
`recurringInsert` primitive signal is added to `PRIMITIVE_SIGNALS` + the code's `signals`).

**Explicitly NOT done — and why (this is the load-bearing decision):**
- **Attribution exclusion (dropping the recurring container from the delta) is DECLINED**, per the #30
  in-window decision (`docs/summaries/v0.5-inwindow-attribution-decision.md`): within-window signal
  cannot distinguish a long causal payload (streamed results / SSE) from a background feed, and a
  false-drop (omitting the modal the click opened) is "the one unforgivable error." So #7 FLAGS, never
  DROPS — the delta keeps every node; `droppedBackground` is unaffected.
- **Settle-promptness (resolving settle early on recurring churn) is DEFERRED**, not tuned here. It has
  the SAME unclosable false-early hazard (freezing a slow-streaming payload mid-stream → a silent
  miss), and the #30 doc's own guidance is that revisiting this needs "corpus telemetry, not
  optimism" — i.e. the owner's real apps (#25/#41). Tuning an aggressive settle-early heuristic
  autonomously, with no real-page corpus, would gamble the project's PRIMARY concern (accuracy) for a
  secondary benefit (speed). Settle timing is therefore byte-unchanged.

So #7 ships its accuracy-SAFE core: it DETECTS + EXPLAINS the churn (completing the residual named at
`observer.ts:250` — the pre-arm baseline misses churn that starts after the action) with ZERO change
to settle timing or membership. The default path is byte-unchanged (the flag is absent unless a
non-baseline signature recurs ≥ `RECUR_MIN` AND settle capped).

**Conservative threshold rationale (and its honest residual):** count alone would false-positive on a
large bounded list reveal (4+ similar rows appearing once). The `hitMaxWait` gate reduces that: a
list that appears and quiesces within the cap is NOT flagged. But `hitMaxWait` is not a clean
bounded-vs-unbounded split — it fires whenever settle doesn't quiesce before the cap, for ANY reason.
So the gate really separates "bounded-and-fast-enough-to-quiesce" from "(unbounded OR
slow/large-enough-to-cap)." **Residual (acknowledged):** a legitimate but slow/large in-window
streamed payload — the exact streamed-results/SSE case cited above to justify not dropping — has a
high `recurringInsert` AND caps, so it ALSO trips the `background-churn` (suspected) flag. This is
tolerated precisely because #7 only FLAGS, never drops: the worst outcome is a soft `suspected` note
over a delta that still keeps every node, never a false-drop. Distinguishing a slow causal payload
from a real feed is the same within-window indistinguishability #30 names, and closing it needs the
deferred real-app corpus telemetry — not an autonomous guess.

**Alternatives rejected:**
- Drop the in-window-recurring container from the delta — declined-as-unsafe (#30).
- Make settle resolve early on recurring churn — deferred pending real-corpus telemetry (#30).
- Count-only threshold (no `hitMaxWait`) — false-positives a legitimate large list reveal.
- A new taxonomy code — unneeded; `background-churn` already models this (avoids DW-04 code churn).

**Affected areas:** `src/injected/observer.ts` (`winInsertCount` tally + collect report),
`src/host/types.ts` (`CollectResult.recurringInsert`, default-absent `DeltaStats.recurringInsert`),
`src/host/actAndObserve.ts` (thresholded, non-behavioral flag), `src/host/diagnose.ts` (`RECUR_MIN`,

**Review-hardened (1 MEDIUM + 1 LOW, fixed before merge):**
- *(MEDIUM) DW-03 over-claim* — an earlier draft called the `hitMaxWait` gate "the honest

## 2026-07-14 — Add the `input-not-committed` taxonomy code (v0.9 Move 1: post-settle input-integrity)

**Decision:** Add ONE new root-cause code `input-not-committed` in a NEW category
`outcome-integrity`, grounded in a NEW primitive signal `committed-value`. It fires — always
`suspected` — when a value-bearing action (`fill`/`type`) reported success but the field's
**post-settle committed value is a shorter subsequence of the intended value AND at least one
DROPPED character is a letter or number** (real content lost): `never-committed` (empty),
`truncated` (a proper prefix), or `dropped-keystrokes` (a non-prefix subsequence). Two mask classes
are **deliberately not flagged** (ambiguous mask-vs-corruption; DW-03 "unsure beats confidently
wrong"): a case/reorder transform (not a subsequence at all), AND a purely subtractive
separator/whitespace mask (`4111 1111` → `41111111`) which IS a shorter subsequence but drops only
formatting characters. This is the first DW code in the "the action succeeded but its OUTCOME is
wrong" class — distinct from every existing code, which is about whether the action was *blocked* or
whether the *delta membership* is right.

**Context:** v0.9 reframes DW as the honest cross-class cause-classifier for agents. An offline run
of v0.8 over 216 real nightly traces showed DW names a cause on only ~10% of failures and ~44% are
bare unattributed timeouts — the growth target. A kill-hypothesis POC (7 synthetic legacy widgets ×
{fill, pressSequentially}) confirmed the gate: **the drop lands AFTER Playwright's synchronous
post-fill check returns success.** A debounce-then-clear SuggestBox (the #41 pathology) and a
deferred-truncate field both left `immediate == intended` but `committed != intended` — Playwright's
`fill`/`type` returned success with a silently-wrong field, which its synchronous check structurally
cannot see. A `pressSequentially`-only subsequence drop (a bulk `.value` set survives) confirmed the
typed primitive is required. Pure value-comparison ⇒ language- and obfuscation-independent (no
role/name/text) — a bullseye for the legacy/non-English target. No tool on the market detects it.

**Grounding + confidence:** the signal is `committed-value` (a post-settle read of `el.value` vs the
caller's intended value). Confidence is clamped to `suspected` by design and never `confirmed`: the
comparison is a hypothesis about intent, not an authoritative bug verdict — a deferred change can be
a legitimate reformat, and the caller's typed string is not guaranteed to be the intended *final*
state (DW-03). The diagnosis never says "Playwright's fill failed" (Playwright succeeded; the widget
mutated the value after) — it says "the field committed a different/shorter value after the action."

**Privacy:** the raw intended/committed strings can be a password or PII, so they NEVER enter the
Delta/stats/serializer. Only `{ shape, intendedLen, committedLen }` is stored and the detail is
phrased by length + shape ("typed 16 chars, the field committed 8 — a prefix"), never echoing the
value.

**Non-goals kept:** read-and-compare only — DW never re-types, corrects, or retries (the field is
observed at the existing post-settle read point, ~zero added latency). Additive + opt-in: the
`inputIntegrity` option is off by default, so the default delta/stats/serialized bytes are unchanged;
`stats.inputIntegrity` is present only when the option is set AND the value drifted.

**Alternatives rejected:**
- *Five separate codes (one per sub-shape).* Rejected: the agent's route is the same for all three
  loss shapes ("the value didn't stick — re-enter / investigate the field"); the sub-shape belongs in
  the `detail`, like `geom-disagreement` carries its direction. One code keeps the closed taxonomy lean.
- *Flag transforms as a sixth `input-transformed` code.* Rejected for v0.9.0: a mask is legitimate
  behaviour; flagging it (even `suspected`) is the exact "agent learns to ignore it" false-positive
  the make-or-break ≤5% bar warns against. The subsequence-and-shorter relation is the unambiguous
  data-loss signal; the transform case stays honestly unlabeled.
- *Re-use `membership-attribution` instead of a new category.* Rejected: that bucket is about whether
  the reported node SET is right; input-integrity is about the value OUTCOME. Forcing it in would be
  dishonest. A new `outcome-integrity` category is the accurate home.
- *Store the raw values / assert equality in Playwright.* Rejected on privacy (see above) and because
  Playwright's `fill` deliberately does not re-assert the post-async value — that gap is the niche.

**DW-04 process (in order):** (1) this ADR; (2) corpus relabel — `input-not-committed-pos` (delta,
shape `truncated` → hit) + `input-not-committed-confuser` (delta, shape `transformed` → must stay
`unknown`, encoding the FP-guard) in `bench/flake-corpus/cases.ts`; (3) accuracy-harness re-run
(`npm run bench:accuracy` — the three floors stay green: the code is `suspected` so it never touches
confirmed-precision; the pos case is a `hit` so silent-miss is unaffected); (4) THEN update the
frozen SHA lock in `test/taxonomy.spec.ts` and the `CATEGORIES` list.

**Affected areas:** `src/host/taxonomy.ts` (code + category + signal + table), `src/host/types.ts`
(`InputShape`, `InputIntegrityStat`, `DeltaStats.inputIntegrity`), `src/host/input-integrity.ts`
(new — the pure `classifyInput`), `src/host/actAndObserve.ts` (`inputIntegrity` option + post-settle
read), `src/host/diagnose.ts` (emit for loss shapes), `src/mcp/session.ts` (`type` action + thread
the option), `docs/specs/v0.6-root-cause-taxonomy.md` (new section), `bench/flake-corpus/cases.ts` +
`test/taxonomy.spec.ts` (relabel + lock), `test/input-integrity.spec.ts` + fixture + `demo:gwt` (new).

**Review-hardened (2 MAJOR + 1 MINOR from an independent 2-lens adversarial review, all fixed before
merge):**
- *(MAJOR, correctness+honesty) subtractive masks were false-flagged as loss* — the first design
  keyed the loss on "committed is a shorter subsequence of intent," claiming a mask "is not a
  subsequence." That is FALSE for the most common real masks, which REMOVE characters: strip-spaces
  on a card field (`4111 1111` → `41111111`), strip-dashes on a phone, trailing/leading trim. Those
  ARE shorter subsequences, so a legitimate reformat was flagged `input-not-committed` — exactly the
  ≤5% false-positive make-or-break the plan warns about, and the uppercase confuser only escaped
  because case-change breaks the subsequence relation (a strawman). Fixed: a loss now requires a
  DROPPED **letter or number** (`\p{L}\p{N}`, Unicode-aware so a non-English letter still counts on
  the legacy target); dropping only separators/whitespace is `transformed` and unflagged. Locked with
  unit + live strip-spaces/trim cases.
- *(MINOR, privacy) the MCP action label leaked the raw value* — the integrity stat stores only
  lengths, but `describe()` baked the plaintext value into `delta.action` (`fill "#pw" = "hunter2"`),
  which `render()`/serialize print — contradicting the value-never-persisted privacy stance for the
  exact field it protects. Fixed: `describe()` redacts value-bearing actions to `<N chars>`.

## 2026-07-14 — v0.9 Move 2: routing signals are co-occurrence metadata, NOT taxonomy codes

**Decision:** The offline ownership-routing arm (`diagnose-trace`) reads the trace's OWN in-page
`console` (error/warning) and uncaught `pageError` events, correlates them to the failing action's
time window `[before.startTime, after.endTime]`, and surfaces them as **routing signals** — adjacent
metadata in the report, deliberately OUTSIDE the closed root-cause taxonomy. It emits a single
`suspectedNotDomCause` hint that flips ONLY when an uncaught `pageError` co-occurred AND Deltawright
named no DOM cause (it stayed `unsure`). No new `RootCauseCode`, no `Diagnosis`, no verdict.

**Context:** v0.9 reframes DW as the honest cross-class classifier/router. The empirical picture (216
real nightly traces) is ~90% `unsure`, and a large slice of those are not DW's DOM-actionability
class at all — assertions, app-JS faults, backend/setup. Telling the agent "this isn't mine — route
it" is as valuable as naming a cause. The trace already carries the signal (`console`/`pageError`
events, verified by a recon probe); the v0.8 reader simply dropped it.

**Why NOT a taxonomy code:** a co-occurring `console.error`/`pageError` is CO-OCCURRENCE, not
causation — a client-side echo cannot prove a backend/app fault *caused* the failure (DW-03). Minting
`app-js-error`/`request-failed` codes would (a) invite an agent to read them as the determined cause,
(b) drag every routing signal through the DW-04 governance the closed taxonomy reserves for
grounded-in-a-primitive causes, and (c) let a noisy legacy console (which logs errors constantly)
manufacture confident-looking labels. Keeping routing as clearly-labeled adjacent metadata is the
honest home. Only `pageError` (a real uncaught exception) flips the hint; `console.error` is context.

**Honesty guards (all tested):** co-occurrence phrasing everywhere ("co-occurred in the action
window … not proof of cause"); list-AND-clamp (cap at 6 signals, report the true in-window count);
window-correlation excludes out-of-window events; a named DOM cause suppresses the hint (the JS error
becomes context, not a route); the section is ADDITIVE (a clean trace's report is byte-unchanged).

**Scope / deferred:** IN-PAGE signals only (console + pageError), which are universal across raw and
`@playwright/test` traces (the reader merges all `*.trace` members). `test.trace` `stderr`/`stdout`
(the harness/backend-504 channel from the recon) is NOT included — the sampled `@playwright/test`
fixture carries only a redundant test-level `error` event, and whether the real nightly corpus logs
backend faults to `stderr` is unvalidated. A follow-up once that's confirmed. The **live arm**
(`page.on('pageerror'|'console'|'requestfailed'|'response')` bracketing `actAndObserve`) is a
separate PR; this ships the offline arm, the higher-ROI half on a timeout-heavy corpus.

**Alternatives rejected:**
- *A `not-a-dom-cause` taxonomy code.* Rejected: it reads as a determined cause and needs DW-04
  governance for what is fundamentally a co-occurrence hint. The `suspectedNotDomCause` flag +
  recommendation string carry the same routing value without over-claiming.
- *Surface all console levels.* Rejected: log/info/debug is pure noise; only error/warning is kept.
- *Whole-trace correlation.* Rejected: an error in an unrelated action would mis-attribute. The
  window is `[startTime, endTime]`; when an edge trace lacks `endTime` the right edge falls back to
  the NEXT action's start, and an unbounded window (a terminal action with no endTime) never flips
  the hint — so a later unrelated exception can never manufacture a route.

**Affected areas:** `src/trace/read-trace.ts` (capture `console`/`pageError` co-events + action
`endTime`), `src/trace/routing.ts` (new — pure `deriveRouting`), `src/trace/diagnose-trace.ts`
(compute + render the routing section), `test/trace-routing.spec.ts` (new).

## 2026-07-14 — v0.9 Move 3: network-idle quiescence (the live-path pivot), framework-agnostic first

**Decision:** Add an opt-in `awaitQuiescence` to `waitForSettle` (via `actAndObserve`). The injected

reads a framework's OWN idle hook when present (`Ext.Ajax.isLoading()`). When `awaitQuiescence` is set,
settle resolves only once the DOM is quiet AND the app is network-idle, still bounded by `maxWaitMs`;
the result carries `quiescent` (false at a cap = the app was still requesting). Read-only — it never
fires events, forces loads, or blocks a request. Default off → the settle logic + stats are byte-unchanged.

**Context — this is the live-path pivot.** The v0.9 offline arms (Move 1 offline, Move 2 routing) were
measured at only **+1 point** on the real ICE corpus: its opaque-timeout failures are OFFLINE-OPAQUE
(the trace records no cause — see the offline-ceiling assessment). Offline reconstruction cannot recover
a cause the trace never held, so the remaining value is LIVE: acting/observing WHEN the app is actually
ready. This is the plan's Move 3, reframed to lead with the framework-AGNOSTIC win.

**Why framework-agnostic FIRST (not the GWT/JSF adapters):** the plan gated Move 3 because GWT — DW's
only validated stack — is the worst case (a zero-network `Scheduler`, no public idle API → a GWT-only
adapter degrades to a blind counter). But GWT-RPC, ExtJS, and JSF all issue **XHR/fetch**, so an
accurate in-flight counter (the app's REAL network state, monkey-patched — not a guess) catches their
in-flight work generically, with framework hooks as best-effort icing. That sidesteps the gate: the
value doesn't depend on a fragile per-framework idle API. (The counter is accurate for requests made
THROUGH the patched globals — not a heuristic — but it does not see a fetch called via a reference
captured before the patch, or a child frame's own globals; stated honestly, not "the app's real state".)

**Honest scope (stated in code + this ADR):** (1) this improves the observe-consequences niche (a
locator-free settle that waits for real network idle) — the #41-validated DW win; it does NOT fix
actionability timeouts (covered glass / backend-never-renders — those are Playwright's verdict / Move 2).
(2) It does NOT catch GWT's zero-network `Scheduler` deferred waves (not network — that is `#49`'s
late-watch). (3) The patch is installed at inject time (before the action), so it counts
action-triggered requests; requests already in flight during page load before inject are not counted —
acceptable, the action window is what matters. (4) A long-poll / persistent request keeps the count > 0
and settle waits to the cap — honest, bounded, and reported via `quiescent=false`.

**Non-goal guard:** `isQuiescent()` is a read-only `isLoading`/count read; it never `fireEvent`/`setValue`
/force-loads, and settle stays bounded by `maxWaitMs`. No taxonomy code — Move 3 improves the settle
SIGNAL, it does not name a cause.

**Alternatives rejected:**
- *A blind XHR-count guess or a fixed extra sleep.* Rejected: the monkey-patched count is the app's
  ACTUAL in-flight state, not a heuristic — the whole point (and what killed the old blind-counter idea).
- *A GWT-specific Scheduler adapter first.* Rejected: GWT has no public idle API (the gate); the
  network counter already catches GWT-RPC, and the zero-network waves are #49's job.
- *Make it the default.* Rejected: it can extend settle latency on request-heavy pages; opt-in keeps the
  default path byte-unchanged and lets the caller choose the observe-when-ready trade-off.

**Affected areas:** `src/injected/observer.ts` (in-flight counter + `isQuiescent` + `enableQuiescence`
+ `waitForSettle` integration), `src/host/types.ts` (`SettleOptions.awaitQuiescence`,
`SettleResult.quiescent`, `DeltaStats.quiescent`, `DeltawrightApi.enableQuiescence`/`isQuiescent`),
`src/host/actAndObserve.ts` (opt-in `enableQuiescence` before the action + thread the option/stat),
`test/quiescence.spec.ts` + `test/fixtures/quiescence.html` (new). Follow-up: wire it into the
`wait`/`observeConsequences` path (where locator-free settle benefits most) + a real ExtJS/JSF validation.

**Review-hardened (2 MAJOR + 1 MINOR from an independent 2-lens adversarial review, all fixed before
merge):**
- *(MAJOR, non-interference) the patch was installed unconditionally* — even a default, non-opt-in run
  replaced `window.fetch`/`XHR.prototype.send` (output byte-unchanged, but the page's native globals
  were tampered — a bot-detection / anti-tamper layer could notice). Fixed: the counter is now installed
  via `enableQuiescence()`, which the host calls before the action ONLY when `awaitQuiescence` is set —
  so the default path leaves the native globals genuinely untouched.
- *(MAJOR, robustness) a synchronous `XHR.send()` throw leaked the counter* — `inFlight++` ran before
  `send()`, so a `send()`-before-`open()` (`InvalidStateError`) incremented but never decremented (the
  request never dispatches → no `loadend`), wedging `isQuiescent()` stuck-false for the page's life and
  burning every future opt-in settle to the cap. Fixed: call through FIRST, increment only after a
  successful `apply` (same reorder for `fetch`).
- *(MINOR, honesty) "the app's real in-flight state, not a guess" over-claimed* — it counts requests
  through the patched globals, not a `fetch` captured before the patch or a child frame's globals.
  Reworded across the code + this ADR.

## 2026-07-15 — v0.9 Move 2 LIVE arm: page.on(...) ownership-routing bracketing actAndObserve

**Decision:** Ship the **live** half of Move 2, promised as "a separate PR" by the offline-arm ADR
(2026-07-14). Opt-in `routeSignals` on `actAndObserve` attaches four page-level listeners —
`response` (kept only when status ≥ 400), `requestfailed`, `pageerror`, `console` (kept only
error/warning) — just before the action and detaches them in a `finally` after settle closes. The
listener LIFETIME **is** the window (action + settle), so no timestamp math is needed (unlike the
offline arm, which correlates trace-relative `time`s). A new pure builder `buildLiveRouting(collected,
{domCauseNamed})` (`src/host/live-routing.ts`) turns what fired during the window into a
`LiveRoutingReport` surfaced on `stats.routing`. Same honesty rules as the offline arm (DW-03): only an
uncaught `pageerror` flips `suspectedNotDomCause`; a 4xx/5xx response OR a failed request flips
`suspectedBackendCause`; a `console.error` is context, never a verdict-flip. NO taxonomy code, NO
`Diagnosis`, NO verdict change (DW-02) — routing stays adjacent metadata (per the merged Move 2 ADR, so
no DW-04 governance is triggered).

**Context — why the LIVE arm is the higher-value half.** The offline arm is capped by what a trace
records: on the ICE nightly corpus a legacy GWT app that swallows its own JS exceptions leaves the
in-page channel EMPTY, and its backend faults are not logged as machine-detectable HTTP-status lines
(see the offline-ceiling assessment — offline routing moved combined triage just +1 pt). The live arm
catches responses / pageerrors / failed requests **as they happen** through `page.on(...)`, so it
yields on the LIVE path even when a trace would be silent — the same reframe (DW as the honest
cross-class classifier/router) delivered where the signal actually exists.

**Additive + opt-in, default path BYTE-UNCHANGED (the whole point).** Everything gates on the
`routeSignals` boolean, placed exactly where `inputIntegrity` lives on the options. When it is OFF:
`attachLiveRouting` is never called (ZERO listeners attached — no page tampering, mirroring the Move 3
non-interference fix that a prior review flagged), and `stats.routing` is absent (the field is
default-absent, mapped `...(routing ? { routing } : {})` exactly like `inputIntegrity`/`quiescent`), so
the delta/stats are identical to today. A dedicated test asserts both halves (no field AND
`page.listenerCount(...)` unchanged around the call).

**No leaks (scoped attach/detach).** The listeners use named handler refs and are removed with
`page.off(...)` (never `removeAllListeners`, which would nuke unrelated listeners) inside a `finally`
wrapping the action + settle read — so a THROWING action (a Playwright timeout) still detaches them and
the page's `listenerCount('response'|'requestfailed'|'pageerror'|'console')` returns to its pre-call
baseline. Tested both on the success path and on an action that rejects.

**`domCauseNamed = false` in the live wiring (honest by construction).** The report is built only after
the action reached settle THROUGH Playwright — i.e. its DOM action outcome was SUCCESS, so Deltawright
named no DOM-actionability cause. Had the action failed on one, Playwright would have thrown and the
report would never be built (we re-throw). So a co-occurring uncaught `pageerror`/backend error may
legitimately flip the route-elsewhere hint ("the click worked; if the higher-level goal still failed,
it is probably not a DOM-actionability issue"). The pure builder keeps `domCauseNamed` a parameter so a
caller/reviewer that DID name a DOM cause can pass `true` to suppress the flip (co-events stay context).

**PRIVACY (runtime output to the caller).** Console text and response URLs can carry PII, so the report
NEVER surfaces a raw URL or full console text — only a kind, a status code, a **query-stripped** URL
path (`origin + pathname`), and a **length-capped** snippet (first 200 chars + a `(N chars)` length
indicator). Mirrors the redaction stance elsewhere (input-integrity surfaces lengths/shape; MCP labels
redacted). A test asserts a `?token=…&pii=…` query and a long message never appear verbatim in the
serialized report.

**List-and-clamp (mirrors the offline arm).** Signals are capped at `MAX_SIGNALS = 6` with the true
pre-cap total in `windowCount`, and the flip-evidence `pageerror` is kept FIRST so the clamp can never
slice out the signal the recommendation cites. `pageErrorCount` and a scalar `backendCount` are always
reported regardless of the cap.

**Parallel, not coupled.** The live report is a PARALLEL host type/module (`src/host/live-routing.ts`),
not a reuse of the frozen offline `src/trace/routing.ts` (untouched). Field names are kept consistent
(`signals`, `windowCount`, `pageErrorCount`, a backend count, `suspectedNotDomCause`,
`suspectedBackendCause`, `recommendation`) and the recommendation uses the SAME phrasing style, but the
live arm does not depend on trace parsing. The pure builder is separated from the impure
listener-collection so it is unit-testable without a browser (7 pure-builder tests).

**Alternatives rejected:**
- *Reuse `deriveRouting`/`RoutingReport` from `src/trace/routing.ts`.* Rejected: that arm is
  time-window-correlated over trace co-events; the live arm's window is the listener lifetime and its
  backend channel is HTTP responses/failed requests, not stdout/stderr lines. Coupling them would drag
  trace-parsing types into the hot `actAndObserve` path for no gain. A parallel module keeps both frozen
  and consistent.
- *Surface the full URL / console text.* Rejected: leaks query-string tokens and PII; a status +
  stripped path + capped snippet carry the routing value without the leak.
- *Attach listeners unconditionally and only gate the output.* Rejected: that tampers with the page's
  event surface on the default path (the exact non-interference regression a Move 3 review caught). Gate
  the ATTACH, not just the report.
- *Flip a verdict / mint an `app-js-error` code.* Rejected for the same reasons as the offline arm —
  co-occurrence is not causation; routing stays metadata (no DW-04 governance).

**Affected areas:** `src/host/live-routing.ts` (new — `LiveRoutingReport`/`LiveRoutingSignal`/
`CollectedLiveSignals`/`RawLiveSignal`, pure `buildLiveRouting`, impure `attachLiveRouting`),
`src/host/actAndObserve.ts` (opt-in `routeSignals`: attach before the action, detach in a `finally`,
build + conditionally surface `stats.routing`), `src/host/types.ts` (`DeltaStats.routing?`),
`src/index.ts` (export the report types + builder), `test/live-routing.spec.ts` +
`test/fixtures/live-routing.html` (new).

**Tags:** `routing` `live-path` `DW-02` `DW-03` `privacy` `v0.9` `move-2`
