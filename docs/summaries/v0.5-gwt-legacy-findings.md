## The correction that shaped this

The tempting pitch — "Playwright fails on GWT, Deltawright fixes it" — **does not survive
scrutiny**, and shipping it would collapse on the exact pain point it targets. Idiomatic,
web-first Playwright (`await expect(locator).toBeVisible()`, auto-waiting text/CSS
locators) polls straight through GWT's deferred-command cascade and **passes**. The naive
failures a demo could show — `accessibility.snapshot()` racing a deferred render,
`waitForLoadState('networkidle')` as the wrong signal, manual `boundingBox()` coordinate
math reading a pre-reposition rect, `getByRole()` finding no role on a `<td>` — are
**strawmen**: a competent engineer rewrites them in three idiomatic lines and they go
green. So every baseline here is paired with its idiomatic **steel-man**, and only cases
that genuinely defeat idiomatic Playwright are claimed as wins.

## What GWT actually does (reproduced in `gwt.html`)

- **Deferred commands, no network:** `Scheduler.scheduleFinally` (stack-unwind) →
  `scheduleDeferred` (≈`setTimeout(0)`) → layout frame, so widgets attach *after* the click
  promise resolves, with zero network I/O.
- **Glass overlay:** `PopupPanel.setGlassEnabled(true)` inserts a full-viewport
  `div.gwt-PopupPanelGlass` that intercepts pointer events (`gwt-` theme classes are stable,
  unlike obfuscated CssResource app classes).
- **Self-repositioning `DialogBox`**, delegated event dispatch on ancestors, role-less
  hashed-class widgets, a hidden history iframe.

## Findings — three scoped wins, three honest limits

| Case | What Deltawright does | Honest label |
|---|---|---|
| **A** deferred-observe | A **locator-free, network-free** structural-quiescence settle catches the deferred render and returns a stable `data-dw-ref` handle + final rect. A zero-wait read misses it. | ✅ **Genuine win** — but only when *observing consequences* with no target to name; for a known locator, idiomatic `toBeVisible()` also passes. |
| **B** glass overlay | Surfaces `coveredBy=<the covering class>` (stable `gwt-PopupPanelGlass` OR an obfuscated CssResource class — class-agnostic, see B2) as a compact pre-action fact; a sub-second labeled `NOT-actionable(covered)` vs a 120s click-hang. **Calibrated as the real portal's #1 stable actionability flake.** | ⚠️ **Diagnostic + latency, not a correctness fix** — it *agrees* with Playwright (already correct); DW cannot dismiss the glass. |
| **C** disabled/covered | Makes the geometry↔Playwright disagreement visible (`[geom:ACTIONABLE]`) while Playwright's authoritative verdict still ships. | ✅ **Visibility win.** |
| **D** role-less cell | Reports "structure changed here" with ref + rect, `role:null`/`name:null`. | 🔎 **Identification limit — NARROWED by calibration.** The real app has substantial ARIA (`getByRole` is its dominant locator), so this residual applies to **legacy cell-tables only**, not the whole app; for role-less cells its textContent name is the same handle `getByText` already uses. |
| **E** two-wave render | Inter-wave gap > `quietMs` (120 ms) ⇒ settle resolves on wave 1 and **silently** under-reports wave 2 (`hitMaxWait:false`, no suspected-miss line). | 🔴 **Silent correctness gap.** |
| **F** JS-timer recenter | `getAnimations()` is empty for JS style writes, so `settleAnimations` is a no-op and the annotated rect is stale (verdict stays correct; only the geometry annotation). A CSS transition is awaited correctly. | 🔴 **Geometry-staleness gap.** |

## What Deltawright CANNOT do (stated on the same slide as the wins)

- Cannot dismiss the glass or make a covered/disabled control clickable — those verdicts
  are **correct**, and DW must never rescue them.
- Cannot semantically know GWT deferred/RPC completion — structural quiescence is a
  mutation-*timing* heuristic (hence gap E).
- Cannot bound an RPC wait it has no signal for (only lever: raise `maxWaitMs`).
- Cannot keep the annotated rect correct through a JS-timer recenter (gap F).
- Cannot keep a `data-dw-ref` valid across a GWT re-render/detach — it staleifies exactly
  like any Playwright locator.
- The fixture is **hand-authored synthetic DOM**, not compiled GWT — it proves only that
  "Deltawright behaves correctly on a DOM we assert resembles GWT."

## The checksum

`src/host/checksum.ts` fingerprints the delta's structure + semantics (kind, role/tag,
name, verdict, `[geom]` disagreement, size bucket, coverage) while dropping raw pixels,
refs, styles, timing, and record order — and hashing `gwt-*` classes stably while
tokenizing obfuscated app classes (rebuilt each compile). It is a **regression guard
only**: a matching hash asserts output == captured output. It is *not* evidence of GWT
correctness and can pass on a fictional DOM.

## Go / no-go

- **GO** to position Deltawright on legacy GWT as: *a better completion **moment** for
  observe-what-changed (no locator needed), a stable **handle** on the appeared node, and
  an independent **coverage / disagreement second-opinion**.* All three hold in
  `test/gwt.spec.ts`, and the two silent gaps (E, F) are asserted, not buried.
- **NO-GO** to claim Deltawright "fixes Playwright's GWT actionability," to lean on
  strawman baselines, or to attribute the actionability *verdict* to settle (the
  trial-click's own auto-wait gets the verdict; settle only changes *which* nodes enter
  the delta and the annotated rect).

## Calibration to a real portal (2026-07) — the credibility dependency, DISCHARGED (4/5)

The fixture's hypotheses were calibrated against a **real production enterprise GWT portal**
(a large claims-management application; ~290 Playwright tests run nightly). Four of the five
real-trace dependencies this doc flagged are answered from the real suite + nightly failure
data; the fifth (the exact Scheduler primitive + timings) still needs a live DOM/source
trace. **No real-app specifics (URL, test IDs, or actual class names) are reproduced here or
in the fixture — only the structural lessons; the fixture's class names are synthetic.**

**VALIDATED — the core bet holds on a real app:**
- **The glass-overlay-intercepts-click pathology is real and is the single largest STABLE
  actionability failure cluster** — ~13 tests hanging on `locator.click: Timeout 120000ms`
  because a full-viewport overlay `intercepts pointer events`, unchanged night over night.
  That is exactly Case B's target: DW collapses the 120s hang into a **sub-second, labeled
  `NOT-actionable(covered)` + a structured `coveredBy`**. The cluster's size + stability is
  the strongest real-world evidence for the Case-B value proposition.
- **Delegated, cell-based dispatch is real** — legacy widgets render as `<table>`s carrying
  GWT's `__gwtcellbasedwidgetimpldispatchingblur` marker (events on ancestors, not per-cell),
  confirming the fixture's central-delegated-dispatch + role-less-cell model (Case D).
- **Embedded iframes for specific widgets** (rich-text editors) confirm the frame-tree
  concern; DW's same-origin frame arming applies to those widgets.

**CORRECTED — fixture + doc updated:**
- **Overlay classes are a MIX, not one stable `gwt-*` theme class.** The real portal's
  overlays span stable `gwt-PopupPanel`/`gwt-DialogBox`, **obfuscated CssResource** classes,
  AND a newer **`materialPaper`** Portal widget set — and the #1 flake's glass was an
  *obfuscated* class. The fixture now models an obfuscated-glass + materialPaper variant
  (`test/gwt.spec.ts` **B2**), proving DW's `coveredBy` surfacing is **class-agnostic** (it
  reports whatever class covers the target); only the checksum tokenizes the obfuscated class
  (a `gwt-*` one is kept). The earlier "GWT does not obfuscate the glass" assumption was too
  narrow.
- **ARIA roles are SUBSTANTIALLY present** — `getByRole` is the dominant locator strategy in
  the real suite (it outnumbers xpath/obfuscated-class locators). So Case D's "role-less
  identification limit" narrows to **legacy cell-tables only**, not the whole app. The open
  question "whether `aria.client.Roles` is present (which would change Case D entirely)" is
  answered: **partially present** → Case D is a residual, not the norm.

**HONEST ADDRESSABLE SHARE — the boundary this sharpens:** DW targets the *actionability /
observe-consequences* failure class only. In the real nightly, that class (overlay-intercept
+ a couple of related interception cases) was a large **stable** minority; the **majority** of
failures were **backend/environment** (gateway/API 504s at data seed, a document-generation
service outage) or **input reliability** (a GWT SuggestBox dropping keystrokes under load) —
none of which DW addresses (it observes + labels; it does not fix backends or inject input).
So the pitch is unchanged and now quantified: DW converts the *biggest stable actionability*
flake from a 120s hang into a sub-second labeled fact — it does **not** move the environment
or input-reliability clusters that dominate the raw failure count.

**Residual (the 5th dependency):** the exact Scheduler primitive (`scheduleFinally` /
`scheduleDeferred`) call chain + real inter-wave timings still want a live DOM/source trace to
calibrate the fixture's deferral model and any committed golden timing — the synthetic
`queueMicrotask → setTimeout(0) → rAF` chain is a faithful *model*, not a measured one. It
pairs naturally with the #25 real-app benchmark.
