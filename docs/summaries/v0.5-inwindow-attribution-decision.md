# summary · v0.5-inwindow-attribution-decision · 2026-07-08

> Issue #30 remainder — the advanced-attribution "rewrite". Decision: ship the
> **trusted-event anchor as a KEEP-ONLY rescue** (`inWindowRecurrence: true`, default
> off) and **decline independent in-window dropping** as unsafe. Reached via a 4-design
> adversarial panel; every independent design had an unclosable false-drop.

## The problem #30 set out to solve

The shipped slice (#30, PR #39) drops background element-insertions using a **pre-arm
baseline**: a signature (`parentTag>tag.firstClass|role`) that recurs ≥2× in the ~150 ms
before the action is background (a toast / live-feed row / virtualized item) and its added
roots are dropped. It is provably conservative — it only condemns what was already
recurring *before* the action existed — but it costs 150 ms up front and cannot catch
background that only starts churning after the action.

The headline #30 goal was to drop background using signal from **within the settle
window** (the ~120–2000 ms after the action), to remove that latency and widen coverage.

## Why independent in-window dropping is declined

A design panel generated four independent approaches (temporal-cadence, locality+recurrence,
signal-fusion, skeptic-minimal); each was handed to an adversary tasked with constructing a
page where it drops the action's real change. The result was unanimous:

**With only within-window signal you cannot distinguish a long causal payload from a
recurring background feed.** A streamed search result list, a staggered master-detail chat
mount, an SSE generate-stream longer than the 2000 ms cap, and a multi-wave dashboard
refresh all share the exact properties any in-window classifier keys on: same insertion
signature, steady-ish cadence, wide temporal spread, and a tail that is still arriving when
the settle cap fires. `maxWaitMs` is a **timeout, not a semantic boundary**, so
"still-arriving-at-the-cap" carries zero information for any payload longer than the cap.

Every independent design therefore had a **high-danger, unclosable false-drop** — and a
false-drop (omitting the modal the click actually opened) is the one unforgivable error
under Deltawright's core invariant: *when uncertain, KEEP*. A false-keep (leaking some
background) is a tolerable quality bug; a false-drop destroys trust.

Even the weakest safe-sounding variant — "in-window evidence may only **reinforce** a
signature the baseline already saw ≥1×" — breaks, because `insertSig` keys on
`classList[0]`, which in real apps is a shared Tailwind / CSS-module / hashed class. A
count==1 collision between an unrelated baseline insert and the action's genuine output is
common (the "CRM Refresh repopulates the dashboard" case drops the whole payload).

## What we ship instead: the KEEP-ONLY anchor rescue

The one safe, valuable idea from the panel (the locality safety valve, inverted): a
**trusted-event anchor**.

- During the window (only when `inWindowRecurrence: true`), capture the **first**
  `isTrusted` event and latch the action's origin: the target element (via
  `composedPath()[0]`, shadow-piercing) and the click point (`clientX/Y`, or the target's
  center for keyboard/input events).
- In `coalesce`, the baseline's background-insert drop becomes **per-root**: a would-be
  dropped root is **RESCUED** (kept) when it is `isActionLocal` — in the DOM lineage of the
  anchor target, or within `ANCHOR_RADIUS = 64 px` of the click point.

**It is KEEP-ONLY, at the added-root level.** The anchor can only move an added *root*
from *dropped* to *kept*; it never causes a drop and never promotes a signature to
background. So:

- Enabling it **strictly reduces** false-drops.
- It fixes a **latent false-drop already in the shipped slice**: an app that reuses a
  background signature (a toast class) for the action's own confirmation, rendered at the
  click, previously had that confirmation dropped; now it is kept.
- It survives every panel false-drop scenario **trivially**, because none of those payload
  signatures are in `bgInsert` — so none was ever eligible to be dropped, anchor or not.

One precision the adversarial verification surfaced: keep-only is exact at the *root*
level, not literally "no output node ever changes." Rescuing a root R re-scopes R's
descendants into the added subtree (as every added root does), so a *mutating
non-interactive descendant* of R that would have surfaced as a **standalone**
`attrChanged`/`textChanged` while R was dropped is now correctly **subsumed** under R. That
is strictly *more correct* — the shipped (flag-off) path orphans that descendant's change
node under a dropped parent — and is locked in by the "folds its mutating descendants" case
in `test/rescue.spec.ts`. The 64 px geometry fallback in `isActionLocal` can also over-KEEP
a genuine background insert that merely renders near the click; this is a deliberate
bias-to-KEEP (a leaked node is tolerable, dropping the action's output is not).

### Degrade-to-KEEP guarantees (all verified in `test/rescue.spec.ts`)

1. **Flag off (default):** no listeners installed, `anchorLatched` false, the rescue branch
   is unreachable — output byte-identical to the shipped slice.
2. **No trusted anchor** (a `page.evaluate`-driven or script-dispatched `.click()`): the
   event is `isTrusted === false`, the anchor never latches, behavior == shipped slice. We
   never drop *more* when the anchor is absent.
3. **`getBoundingClientRect` throws / detached node:** the geometry check is `try`-guarded
   and falls through to not-local — reproducing a prior drop, never a new one.
4. **Signature not in `bgInsert`** (every adversary payload): unconditionally kept — no
   in-window count, cadence, spread, or geometry can move it to drop.

The operational invariant: *the only roots this feature can drop are exactly the roots the
already-proven baseline filter drops today, minus the ones the anchor rescues.*

## Regression coverage (`test/rescue.spec.ts`, 4 cases)

- **rescue** — a background signature reused for the action's confirmation inside the
  clicked control is kept; the far background instances are still dropped
  (`droppedBackground > 0`).
- **additivity** — with the flag off, that same confirmation is dropped (shipped parity),
  proving the change is purely additive.
- **untrusted-parity** — a script-dispatched click latches no anchor, so behavior matches
  the shipped slice.
- **no-drop streamed** — a 10-item streamed same-signature payload with an empty baseline is
  never dropped (`droppedBackground === 0`).

## What is deliberately NOT shipped

- Independent in-window condemnation (cadence / CoV / tail / group-count / far-from-anchor
  scoring) — declined; each false-drops a real payload.
- Baseline-count ≥ 1 reinforcement — declined; shared-class signature collisions.
- Cutting `baselineMs` — the recovery signal is exactly the unsafe one.

The `inWindowRecurrence` flag stays default-off and baseline-gated. If we ever revisit
independent dropping, the honest path is to gather corpus telemetry (insert timing + add/
remove cycles, verdict-neutral) over the #25 admissible corpus and decide from data — not
optimism. See ADR 2026-07-08 and the design panel record.
