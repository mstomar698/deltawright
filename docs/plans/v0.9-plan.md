# v0.9 plan — from "faster actionability verdict" to "the honest cross-class cause-classifier for agents"

**Status:** PLAN (2026-07-14). Follows v0.8 (`diagnose-trace` shipped). Grounded in an offline analysis
of a **real production enterprise GWT nightly suite** (216 per-test traces, ~118 cases; analyzed
locally — only sanitized aggregate ratios appear here, per the legacy-portal guardrail) plus the #25 /
#41 findings and a design-panel + adversarial-skeptic research pass.

---

## 0. Why v0.9 exists — the honest empirical picture

Running the shipped v0.8 classifier **offline over 216 real nightly traces** gives the number that
defines this milestone:

- **DW names a specific actionability cause on ~10% of real failures** (covered-by-overlay + not-visible).
- **~90% it correctly returns `unsure`** — "not my class." That 90% splits roughly into:
  - **~19% assertions** (`expect …` mismatches) — genuinely not an actionability problem. DW is *right*
    to abstain, and that abstention is itself a routing signal ("this is a data/logic bug, not a DOM one").
  - **~44% unattributed timeouts** (`Timeout … exceeded` with no cause DW's taxonomy recognizes) — **the
    growth target.** Some fraction of these are input-drop, a late/deferred wave, or an overlay whose class
    DW didn't key. Converting even half of this bucket into named causes is the whole v0.9 prize.
  - **~27% other** (page-closed, navigation, framework errors).

Two facts from the same recon reshape the design:

1. **The backend-5xx cluster is NOT in the page trace.** `*.network` members here are resource-snapshots,
   not HTTP-status events, and the #41 gateway-504s happen in **test-harness seed / setup HTTP calls**,
   outside the browser trace. The offline backend signal lives in **`test.trace`'s `error` / `stderr` /
   `stdout` events**, which the v0.8 reader does not yet read. Move 2 must target those, not `*.network`.
2. **The suite is deeply POM-based** — 272 specs, 176 page objects, base-page wrappers (`click()`,
   `clickSave()`, `fill`, …) over **856 `.click()`, 238 `.fill()`, 73 char-by-char `.pressSequentially()`/
   `.type()`**. So DW can be instrumented in **~6 base-page methods**, not 856 call sites. Low effort,
   suite-wide reach.

**The reframe (the only pitch that survives scrutiny):** stop selling DW as a *faster / second-opinion
actionability verdict*. Sell it as the **at-failure-time, machine-readable cause classifier for an
autonomous agent that has no APM and no human-pre-solved selectors** — one confidence-banded output that
says which failures are DW's *and, just as valuably, which are not* (assertion / backend / setup), so the
agent **routes instead of blindly retrying or self-healing.** a11y-tree agents, vision agents, and
self-healers all go blind on obfuscated / non-English legacy DOM — that seat is empty. This is honest: DW
never fixes; it names classes and abstains loudly when it can't. It stays a **minority-class tool on a
backend-dominated portal** — but a genuinely additive one, and it raises the 10%.

**Non-negotiable ceiling (state it in every demo):** DW never fixes, injects, retries, or overrides
(DW-02/03/04). Every capability below ends at **detect-and-label**. v0.9 makes DW's coverage *wider and
more honest*, never *deeper*.

## 1. The three moves, sequenced

Ordered by (real value × feasibility). Each says which trace-bucket it converts and how v0.8's
trace reverse-engineering is the offline substrate that validates it *without a flaky live run*.

### Move 1 (v0.9.0) — Post-settle input-integrity verdict
**The one genuinely new, market-unfilled, language-independent capability.**

- **What:** after any value-bearing action, once the settle window closes, read the field's *committed*
  state (`el.value`, `validity`, `maxLength`, `document.activeElement`) and compare to the **intended
  value** the caller already holds. Emit a `suspected` sub-class: `dropped-keystrokes` (committed value is
  a subsequence of intent — the #41 SuggestBox pathology), `truncated`, `masked/transformed`,
  `never-committed`, `focus-lost-during-input`.
- **Bucket it attacks:** part of the ~44% unattributed-timeout + silent-no-op bucket, and a class DW
  scores **zero** on today. Suite surface: the **311 fill/type/pressSequentially sites** (the 73
  char-by-char ones are highest-risk).
- **Why real, not marginal:** it catches the **async debounce-then-clear** that Playwright's *synchronous*
  post-fill check structurally cannot see (PW sets `.value`, returns success; the widget re-reads and
  clears it frames later). Pure value-comparison → **language- and obfuscation-independent** (no
  role/name/text) → a bullseye for the target. No tool on the market detects it.
- **v0.8 trace-RE leverage:** the offline path is a `fill`/`type` action's `params.value` (in the trace)
  vs the field's value in the **after `frame-snapshot`** — so `diagnose-trace` can flag input-drop
  **offline across the whole corpus**, no live run. (Live path reads it in `actAndObserve`.)
- **Non-goal guard:** reads & compares only; never re-types/corrects. Clamps to `suspected`. Attribution
  must say "typed X, field now shows Y — suspected input-drop," **never** "Playwright's fill failed"
  (PW succeeded; the widget mutated it after).
- **Effort:** thread intended value through the MCP `fill` path + add a char-by-char typed primitive; add
  `readValue()` at the existing post-settle read point (`actAndObserve.ts` `lateResult`, ~zero latency);
  one taxonomy code (DW-04 governance: ADR + corpus relabel + accuracy re-run) + a validation corpus.

### Move 2 (v0.9.x) — Honest ownership-routing (the reframe's backbone)
- **What:** (a) **Live:** bracket `actAndObserve` with `page.on('response'|'requestfailed'|'pageerror'|
  'console')` for the settle window (**DW attaches zero such listeners today**) and fold events into the
  delta. (b) **Offline:** extend `diagnose-trace` to read the trace members it currently **unzips and
  drops** — `test.trace`'s `error`/`stderr`/`stdout` (the setup/backend signal — see §0), plus in-page
  `console`/`pageerror` and any real request-failure. Emit one banded cause spanning actionability **+**
  app-JS-error **+** in-page request-failed **+** "target absent/covered in the snapshot" **+** the
  routing verdict *not-a-DOM-cause*.
- **Bucket it attacks:** the ~19% assertion + ~27% other buckets get an explicit label instead of "unsure";
  the agent learns *this isn't mine* and routes (escalate / retry-backend / fix-data) rather than burning
  self-heal cycles.
- **The honest cap (build the NEGATIVE framing only):** the value is *ownership routing*, **not causal
  resolution** — APM owns "the 504 caused it," and a client-side echo can't prove causation. Positive
  causal attribution is killed (§2). On RPC-heavy GWT the window is noisy, so this must **list candidates
  and clamp**, never assert *the* cause.
- **v0.8 trace-RE leverage:** this **is** mostly a `diagnose-trace` expansion — the reader/zip layer is
  already general; add event cases + correlation-by-timestamp. Validates offline over the whole corpus.
- **Non-goal guard:** never upgrades a `console.error` to a verdict; never overrides PW's actionability
  verdict. Co-occurrence phrasing + suspected-clamp are load-bearing.

### Move 3 (v0.9.y — a GATED bet, not a green light) — Framework-aware quiescence
- **What:** an `addInitScript` that hooks each framework's *own* idle machinery without editing the app —
  GWT `Scheduler`/RPC in-flight, `Ext.Ajax.isLoading` + store `isLoading`, JSF ajax queue — exposing
  `isQuiescent()` that `waitForSettle` consumes **alongside** structural quiescence. "`waitForAngular` for
  legacy Java frameworks." Improves the settle *signal* itself (core DW), not a bolt-on label.
- **Bucket it attacks:** the deferred/RPC-render slice of the timeout bucket — acting *when the app is
  ready*, the #1 root of legacy flakiness.
- **Why gated:** it is the biggest **market** gap (nobody ships a drop-in external probe — the only
  documented option is editing app source), BUT **GWT — DW's only validated stack — is the worst case**
  (zero-network `scheduleFinally→scheduleDeferred`, no public idle API → a GWT adapter degrades to a blind
  counter). The payoff lands on **ExtJS/JSF, which DW has never validated, has zero fixtures for, and which
  carry real version-drift maintenance risk.** **Do not fund on the market gap alone — prove it on a real
  non-GWT legacy app first.**
- **Non-goal guard:** `isQuiescent()` is a read-only `isLoading`/queue read, never `fireEvent`/`setValue`/
  force-load; still bounded by `maxWaitMs`.

## 2. The "killed" ideas, re-examined as sub-components (per request)

The skeptic killed these as **standalone products** (commoditized, or the wrong tool). But demoted to
**internal mechanisms of a surviving move**, several add real value. This is the "pick a better choice that
helps the skeptics in one or more ways" analysis.

| Killed idea (as a product) | Reused as a sub-component of… | The value it adds (honestly) |
|---|---|---|
| **Selector self-healing** | Move 2 routing | Don't auto-heal. But when DW flags `selector-rot`, run the existing `suggest()`+`verifySuggestions()` and **hand the agent a verified alternative handle** ("this selector no longer resolves; a candidate that resolves NOW is …"). Observe-and-explain (suggest, never apply). Turns a dead "not-found" into an actionable route. |
| **Generic log-based RCA** | Move 2 offline | Don't compete with commercial log-RCA on rich CI logs. But read the trace's **own** `error`/`stderr`/`console`/`pageerror` events (the recon shows they exist) to produce the *routing* label. Uses only what's already in the artifact — no new pipeline. |
| **Durable structural locators / cross-render re-anchor** | Move 1 + Move 2 dedup | Killed as a *first-identification* product (collapses to "N candidates, unsure" on role-less cell-tables). But **re-identification of an already-observed node is exactly what Move 1 needs** — re-find the same field after settle to read its committed value. And a stable structural fingerprint makes the flake-dashboard dedup better. Valid internal mechanism, not a product. |
| **Positive causal backend attribution** | Move 2 (negative framing only) | "504 CAUSED it" stays dead (APM-redundant, causally unknowable client-side). The survivor: surface "a request/error co-occurred in this window" as a **candidate the agent weighs**, phrased as co-occurrence, never causation. |
| **Blind XHR in-flight counter** | Move 3 (the non-blind version) | The counter stays dead (stale — `#49` late-watch already catches the zero-network GWT late wave better). But the *intent* — "know when async work is done" — survives as Move 3's framework-quiescence, which reads the app's **own** idle state instead of guessing from network. |

Net: no killed idea returns as a product; three return as **plumbing that raises a surviving move's value.**

## 3. v0.8 trace reverse-engineering as the v0.9 substrate

v0.8 `diagnose-trace` is not just a feature — it's the **offline validation engine** for all of v0.9:

- **Reproducible, non-flaky evidence.** Every move can be validated by running the extended classifier over
  the **existing 216-trace corpus** (and any CI trace archive) offline — no live browser, no flake, no
  reproduction. This is how we prove Move 1/2 raise the 10% *before* touching a live site.
- **Move 2 IS a trace-reader expansion** (add `test.trace` error/stderr + console/pageerror event cases).
- **Move 1 has an offline arm** (`fill.params.value` vs the after-snapshot field value).
- **The final PW-vs-DW-vs-combined report (§6) is produced entirely offline from traces.**

Concrete v0.9 reader work: extend `read-trace.ts` to also surface `error`/`stderr`/`stdout`/`console`/
`pageerror` events and (for Move 1) the after-`frame-snapshot` field values; keep the version-guard and the
dep-free zip; keep every new signal `suspected`.

## 4. Passing-suite instrumentation & the enhancement score

**Question:** even for tests that PASS today, is there value in instrumenting them with DW? **Estimate:
yes, meaningful triage-quality value — but bounded, and NOT a pass-rate change.**

**Why it's cheap:** the suite is POM-based with base-page wrappers. Instrument **~6 base methods**
(`click`/`fill`/`pressSequentially`/`selectOption`/`check`/`navigate`) once and all **856 clicks / 311
typed inputs** across 272 specs inherit it. No per-test edits.

**What instrumentation buys (per failure class, from the 216-trace distribution):**

| Class | Share of failures | What DW-in-the-PO adds | Kind of gain |
|---|---|---|---|
| covered/overlay + not-visible | ~10% | 120 s blind hang → labeled `covered-by-overlay` in ~1.5 s + a durable fingerprint | **triage speed + a named cause** |
| input-drop (Move 1) | part of the ~44% timeout bucket | proactively flags typed-vs-committed drift on the 311 input sites — **catches the SuggestBox flake the run before it tips into a 120 s failure** | **early warning (new class)** |
| assertions | ~19% | explicit "not-a-DOM-cause — data/logic" routing label instead of a bare diff | **routing** |
| backend/setup (Move 2, test-trace) | part of the ~27% | "not-mine — setup/backend" route, so the agent/CI doesn't self-heal a non-DOM failure | **routing** |

**The honest score:** on a **triage-usefulness** axis (does the report name the cause and route it?), DW
instrumentation is a **material uplift** — today ~90% of failures are bare `Timeout … exceeded`; with M1+M2
a large share of that gets a class + a route. On a **pass-rate** axis the score is **zero by design** — DW
never makes a flaky test pass. So the enhancement is: *"same pass/fail, dramatically better failure reports
+ a new proactive input-drop early-warning,"* at ~6 base-method cost. That is the number to sell, and it is
honest. (A quantified before/after is exactly what §6's report produces.)

## 6. The final deliverable — PW vs DW vs combined flake report (offline, over the corpus)

Produced entirely offline from the 216 traces (extended `diagnose-trace`):

- **PW-alone column:** the raw terse error (`Timeout … exceeded`) + the buried call-log.
- **DW-alone column:** the banded cause or an honest `unsure/not-mine`.
- **Combined column:** PW's error + DW's class + route — the hypothesis being that **combined > either
  singular** (PW says *what broke*; DW says *which class it is and whether it's DW's to own*).
- **Verdict metrics:** for how many of the 216 does each column yield an *actionable* triage (named cause OR
  a confident route) vs a bare timeout; how much the combined column reduces "bare-timeout" triage.

**Measured BASELINE (current 0.8 DW, offline over 215 failed nightly traces — the "before"):**

| triage at a glance (route it *without* scraping the call-log) | share |
|---|---|
| PW terse error alone | **19%** (41/215 — mostly the self-explanatory assertions) |
| DW classified (taxonomy) alone | **10%** (22/215) |
| DW surfaced the buried cause line (PW's headline hides it) | 16% (35/215) |
| **PW + DW combined** | **31%** (66/215) |

**So combined (31%) beats PW-alone (19%, ×1.6) and DW-alone (10%, ×3) — the "combined > singular"
hypothesis holds empirically.** The honest gap: **69% is still opaque even combined**, and it's dominated
by the **44% bare-timeout bucket** (95/215) that names no cause today — precisely what Moves 1/2/3 must
convert. This is the number every v0.9 milestone is measured against; re-run per move to show the delta.

This report is the empirical scorecard for the whole reframe. Local + sanitized; a repo-safe version only
after confirmation.

## 6b. The three cross-cutting deliverables — do them NOW, or after a better DW?

All three are planned items. But they are **not** all "wait until the moves ship" — one is most valuable
*today* as the baseline that decides whether v0.9 is even worth funding.

| Deliverable | Do it NOW? | Why / what it buys at each stage |
|---|---|---|
| **Traces flake-report (PW vs DW vs combined)** | **YES — now, then re-run per move.** | With *current* 0.8 DW it is the **BASELINE scorecard**: it quantifies exactly where DW stands (~10% named, ~90% `unsure`) and establishes the "before" that Move 1/2 must beat. It's offline, cheap, reproducible, and it is the honest measurement that justifies — or kills — the whole reframe. Re-running it after each move is how we *prove* the move raised coverage. Running it only "when DW is better" would throw away the baseline. **This is the single most useful thing to do right now.** |
| **Passing-suite enhancement estimate** | **The NUMBER now; the instrumented run later.** | The *estimate* (§4) is deliverable today from the suite surface (856 clicks / 311 inputs / ~6 base-PO wrappers). Actually **instrumenting + measuring** the passing suite with current DW only buys the ~10% actionability labeling — real but thin; its full value needs Move 1 (input-integrity on the 311 inputs) and Move 2 (routing). So: publish the estimate now; run the instrumented before/after **after Move 1** lands. |
| **GWT demo (synthetic + real-site)** | **Synthetic per-move; real-site deferred.** | Tier A (synthetic fixture) is the **acceptance test for each move** — build it *with* the move, not before (there's nothing new to demo on the synthetic fixture with current DW beyond what #41 already showed). Tier B (real-site live) should wait until Move 1/2 exist — demoing current DW on the real portal just re-shows #41's overlay result. Running it now = no new signal. |

**Verdict for "is it helpful right now":** the **traces baseline report — yes, run it now** (and it's mostly
done from the recon). The other two — capture the *estimate/design* now, execute the *measurement/demo* once
Move 1 (and for routing, Move 2) exist, because that's when they produce a signal current DW can't.

## 7. Sequencing, risks, ceiling

- **0.9.0 = Move 1** (input-integrity: live + offline arms + synthetic demo + validation corpus).
- **0.9.x = Move 2** (ownership-routing: trace-reader expansion + live listeners; the killed-idea
  sub-components ride here).
- **0.9.y = Move 3** — **gated**; only after a non-GWT (ExtJS/JSF) quiescence POC proves out.
- **The report (§6)** can land as soon as Move 2's reader expansion exists (it needs the console/error events).
- **Risks:** M1 false-positive rate on GWT's deferred-render base rate (the make-or-break — ≤~5% on clean
  fields or the agent learns to ignore it); M2 correlation-mis-read-as-causation on noisy legacy console;
  M3 version-drift on unvalidated frameworks; DW-04 governance cost per new taxonomy code.
- **Ceiling, restated:** DW stays detect-and-label and minority-class on a backend-dominated portal. v0.9's
  win is **lateral** (name more classes honestly, including which aren't DW's) — never **deeper** (it will
  never make the flaky thing pass). If Move 1's false-positive rate blows out, that is the honest signal DW
  is near its useful limit and should be scoped as an agent-triage tool, not a legacy-enterprise flagship.

## 8. First experiment (proves/kills Move 1)

POC: reproduce the input-drop widget under load + a clean-field control set; thread `action.value`; add
`readValue()`. **Kill-hypothesis:** does the drop land *after* PW's synchronous post-fill check returns
success? (If PW already sees it, Move 1 has no reason to exist — test `fill` AND `pressSequentially`.)
**"Useful" = recall ≥ ~80% of true drops, false-positive ≤ ~5% on clean fields, ~zero added latency, a
sub-label that maps to a distinct agent action.** Secondary (same harness): record how often a backend/JS
error co-occurs in the action window (quantifies Move 2's routing precision before investing).
